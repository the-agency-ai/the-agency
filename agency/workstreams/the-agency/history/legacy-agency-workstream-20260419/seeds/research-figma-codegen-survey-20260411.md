# Research Report: Codegen Tooling Survey — Figma → React / React Native (2026)

Research agent 2 of 4. Angle: codegen tools that take Figma files and emit React / React Native code.

## Summary

- **The market has split into three tiers in 2026.** Tier 1 (viable for production): Builder.io Visual Copilot, Figma Make, Locofy Lightning, Anima. Tier 2 (AI-native, prompt-first, Figma is secondary input): v0.dev, UX Pilot. Tier 3 (toy / demo-grade): everything else including Penpot's Inspect, TeleportHQ, DhiWise.
- **Figma Code Connect is the canonical path forward for design-system fidelity.** Every serious tool in Tier 1 now either consumes Code Connect mappings or ships its own "component mapping" feature that does the same thing. Raw autogenerated CSS+JSX (classic "Dev Mode Inspect") is increasingly a fallback, not a primary workflow.
- **Nobody has solved React Native cleanly.** Builder.io Visual Copilot is the only tool with a credible RN story, and even it hedges between StyleSheet and Tailwind output. Tamagui output is unsupported by all major codegen tools as of April 2026 — this is the biggest gap for monofolk.
- **"AI-native" is mostly repackaging.** v0.dev, UX Pilot, and Anima's AI features all generate prettier code, but they still produce bespoke components that need refactoring against a real design system. The exception is Builder.io, which chains Mitosis (a real compiler) between the AI model and the output.
- **Verdict for the-agency/monofolk:** Our existing `figma-extract` + `figma-diff` pipeline is token-focused and complementary, not redundant. The right next move is a thin adapter for **Figma Code Connect** (the mapping protocol) plus a targeted spike on **Builder.io Visual Copilot** for RN screens. Everything else is prompt-Figma tools we should avoid building on.

## Figma Dev Mode Deep Dive

Dev Mode in 2026 is a different product than "Inspect mode" was in 2023. Three things matter:

1. **Code Connect is the headline feature.** It is a GitHub-backed (or manually mapped) binding between a Figma component and its real code implementation. When a developer opens a Figma component in Dev Mode, instead of seeing auto-generated CSS, they see a snippet rendered from your actual source file. Mapping files live in your repo and are validated by a CLI (`figma connect publish`).
2. **MCP codegen is built in.** Figma ships an MCP server ("enhanced MCP codegen" in their terminology) that lets AI IDEs (Cursor, Windsurf, Claude Code) pull component context directly from Dev Mode without scraping the Figma API. This is the highest-leverage integration point for us — an agent can ask Figma for a component's real code, then adapt.
3. **Multi-framework binding.** Code Connect now supports one-to-many mappings: a single Figma component can map to React, SwiftUI, Jetpack Compose, and Vue implementations simultaneously. For the monofolk (RN + web) case, this means you can ship one Figma library and bind it to both a React and an RN implementation.

**Verdict on Dev Mode:** It is the canonical path for *handoff* but not for *codegen*. It tells developers what the code should be; it does not write new code for new frames. You still need Visual Copilot / Anima / Figma Make / hand-coding to translate a new layout into components. Code Connect makes the handoff faithful; it does not remove the translation step.

## Design-System Integration Analysis

This is where the tools sort themselves out. Three strategies in play:

- **Component mapping (best).** Builder.io Visual Copilot and Figma Code Connect let you declare "this Figma component = this React import path." The generator then emits `<Button variant="primary">` instead of an inline `<div className="...">`. Visual Copilot applies this mapping automatically across new screens once the library is mapped. This is the only strategy that produces maintainable output against a real DS.
- **Token injection (middle).** Anima and Locofy pull your Figma tokens (colors, typography, spacing) into their generation pipeline and emit Tailwind/shadcn code that uses those tokens. The output looks right but the components are still bespoke (per-frame divs, not references to your Button).
- **DS-assumed (weakest).** v0.dev assumes shadcn/ui as the baseline and generates against it. For teams already on shadcn this is fine; for anyone else (MUI, Tamagui, bespoke) the output must be rewritten.

**For monofolk specifically:** Tamagui is not a first-class target for *any* tool. Visual Copilot's Mitosis layer can in theory be extended (Mitosis is open-source and explicitly supports RN+web cross-compilation), but there is no shipped Tamagui target as of April 2026. This is a real gap.

## AI-Native Codegen State of the Art

2026's "AI-native" codegen tools fall into two camps:

- **Pipeline-based** (Builder.io Visual Copilot). AI is one stage in a chain: hierarchy model → Mitosis compiler → framework-specific fine-tune. This is the only approach producing framework-accurate output at scale.
- **Prompt-based** (v0.dev, Figma Make, UX Pilot, Anima AI). A single LLM generates code from designs + prompts. Output is prettier than 2023 tools but still "bespoke translation" — every frame gets regenerated from scratch. Fine for prototyping, weak for a maintained codebase.

The honest assessment: most 2026 AI tools are making the same translation prettier, not solving the codegen problem. The codegen problem is *mapping to an existing design system*, and only component-mapping approaches address it.

## Our Existing Artifacts Compared

Our existing tools occupy a niche these commercial products do not:

- **`figma-extract`** — extracts tokens (colors, fonts) from a Figma file via API plus embedded document data. Writes structured markdown to `claude/knowledge/design-systems/{name}-{version}/`. **This is NOT a codegen tool** — it is a token extractor, and none of the commercial tools do this cleanly into version-controlled, agent-readable markdown. It complements Visual Copilot (feed it our tokens) and Code Connect (bind components separately).
- **`figma-diff`** — screenshots a deployed URL at a viewport and compares to a mockup image. This is visual QA, not codegen. Complementary to everything above.
- **`designsystem-add`** — scaffolding for design system docs. Infrastructure, not codegen.
- **`ordinaryfolk-003/`** — a prior hand-curated extraction. Proves the markdown-first approach works for agent workflows but is a manual artifact.

**Gap we don't fill today:** actual Figma-frame → React/RN code synthesis. We have tokens, diffs, and docs. We do not have a "give me this frame as JSX" path. The commercial tools fill that gap, imperfectly.

**Complementary, not redundant:** Our tools feed the codegen tools. Visual Copilot or Code Connect consumes our extracted tokens; our `figma-diff` validates their output.

## Recommended Short-List

Five tools worth testing against the monofolk codebase, in priority order:

1. **Figma Code Connect** — non-negotiable. Build an adapter in `agency/tools/` that writes Code Connect mapping files from our design-system markdown. Low cost, high leverage.
2. **Builder.io Visual Copilot** — highest-quality RN codegen. Spike: point it at a monofolk screen and measure cleanup effort. If Tamagui adaptation is feasible via Mitosis, this becomes the primary path.
3. **Figma Dev Mode MCP** — connect Claude Code to Figma's MCP server directly. Cleanest path for agent-in-the-loop codegen; no scraping, no rate limits.
4. **Figma Make** — evaluate for principal-facing prototyping only. Do not build a pipeline on it.
5. **Anima** — fallback for web-only flows if Visual Copilot's React (not RN) output disappoints. Strong Storybook generation is a real plus.

Explicitly skip: Locofy (quality issues), v0.dev (not a codegen tool for our use case), UX Pilot (too early), Penpot (not a codegen tool), DhiWise/TeleportHQ (toy tier).
