# DevEx — Product Vision & Requirements

## Vision

DevEx owns the developer experience for agents and principals working in TheAgency. The boundary between "I want to do X" and X actually happening — that's our territory. Test infrastructure, commit workflow, permission model, enforcement tooling, context economics. If it creates friction, we fix it.

## The Problem

TheAgency's commit boundary is broken. The pre-commit hook (`claude/tools/commit-precheck`) runs all 155 tests across 32 BATS files for every commit — including markdown-only changes. It times out. It forces `--no-verify`. Which bypasses QGR enforcement. Which means the quality gate that's supposed to protect every commit is routinely circumvented.

Worse: the tests that DO run can corrupt the live environment. During the ISCP rollout (2026-04-05/06), BATS tests set `core.bare=true` in `.git/config`, leaked 62 test flags into the live ISCP database, and scattered test debris across the working directory. Captain operations were blocked twice in one session. The pre-commit hook re-triggered the corruption in a vicious cycle.

Meanwhile, `settings-template.json` ships too few permissions. Agents prompt for `ls`, `sqlite3`, `git show` — safe, read-only operations. Every prompt breaks agent flow, requires principal intervention, and contradicts the "autonomous by default" principle (Valueflow NFR3).

ISCP fixed the immediate test isolation bugs for its 7 test files. 25 of 32 BATS files still have zero isolation. The Docker test runner only covers ISCP tests. The pre-commit hook has no smart scoping, no timeout, no graceful degradation.

## Users

**Agents** — every agent in the repo hits the commit boundary. A broken pre-commit hook blocks all of them. Missing permissions block all of them. DevEx serves every agent equally.

**Principals** — when agents prompt for safe operations, the principal has to intervene. When the pre-commit hook times out, the principal has to decide whether to `--no-verify`. DevEx reduces principal toil.

**Framework adopters** — when someone runs `agency init`, the permissions and commit workflow they get must work out of the box. DevEx owns the day-zero experience for the commit boundary.

## Requirements

### Functional

| ID | Requirement |
|----|-------------|
| FR1 | **Smart test scoping** — pre-commit detects changed files and runs only relevant tests. Convention-based mapping: `claude/tools/{tool}` → `tests/tools/{tool}.bats`. Package-level fallback for non-mirrored layouts. No mapping = warn and proceed. |
| FR2 | **Non-code skip** — pre-commit skips tests entirely for changes that are exclusively markdown, dispatches, handoffs, transcripts, and other non-code artifacts. Stage-hash delta tolerance: if delta is exclusively `.md` files, allow with warning. |
| FR3 | **Timeout with graceful degradation** — T1 gate budget is 60 seconds (per Valueflow A&D §6). If tests exceed budget, warn but don't block. Log the timeout for telemetry. Never hang the commit indefinitely. |
| FR4 | **Test isolation for all BATS files** — extend ISCP's isolation pattern (`iscp_test_isolation_setup/teardown`) to all 32 BATS test files. Every test file must use isolation helpers. No test may modify live `.git/config`, live ISCP DB, or live working directory. |
| FR5 | **Filesystem pollution guards** — tests that create files in the working tree (`.claude/agents/testname.md`, `claude/agents/testname/`, injection test dirs) must use temp directories, not the live tree. Teardown must verify no debris. |
| FR6 | **Docker full-suite runner** — extend `tests/docker-test.sh` to run all 32 BATS files, not just the 7 ISCP files. Docker is the mechanism for T3 (phase complete) full-suite runs. |
| FR7 | **Permission model audit** — audit all non-destructive agent operations, pre-approve them in `settings-template.json`. Categories: read operations (ls, cat, git show, git log), ISCP tools (flag, dispatch list/read, iscp-check), framework tools (handoff read, agent-identity), and standard file operations on `usr/`, `claude/`, `~/.agency/`. |
| FR8 | **Enforcement registry + audit tool** — YAML manifest (`claude/config/enforcement.yaml`) declaring each capability's ladder position. Audit tool validates: at level N, do all artifacts for levels 1-N exist? Ships alongside the registry — no registry without auditor (per Valueflow A&D §4). |
| FR9 | **Context budget linter** — measure `@` import chain token counts per skill. Warn when a skill injection exceeds 4000 tokens. Ships alongside CLAUDE-THEAGENCY.md decomposition — they must ship together or neither ships (per Valueflow A&D §9). |
| FR10 | **QGR stage-hash enforcement** — `/git-safe-commit` verifies a QGR receipt exists with a matching stage-hash before allowing commit. No receipt = no QG was run = commit blocked. This is the mechanical enforcement for Valueflow FR6. |

### Non-Functional

| ID | Requirement |
|----|-------------|
| NFR1 | **Zero permission prompts for safe operations** — agents must never be prompted for read-only operations on framework paths, ISCP tools, or standard file operations. Measurable: count permission prompts per session, target is zero. |
| NFR2 | **Test hermiticity** — all tests run in isolation from the host environment. No test may modify live git config, live databases, or the working directory outside its allocated temp space. Enforced by isolation helpers and teardown guards. Violations fail the test. |
| NFR3 | **T1 gate under 60 seconds** — iteration commits complete the full T1 gate (stage-hash + format + relevant tests) within 60 seconds. If scoped tests alone exceed 60s, the scoping is wrong, not the budget. |
| NFR4 | **Pre-commit never corrupts** — the pre-commit hook must never run tests that can corrupt the host. All pre-commit test execution uses isolation helpers. Docker is preferred for full-suite; in-process with isolation for scoped runs. |
| NFR5 | **Backward compatible** — existing agents and workflows must not break. New isolation helpers are additive. The pre-commit rewrite must handle projects with no BATS tests (skip gracefully), projects with partial isolation (warn), and projects with full isolation (enforce). |

### Constraints

| ID | Constraint |
|----|-----------|
| C1 | **Build on ISCP's foundation** — `iscp_test_isolation_setup/teardown`, `ISCP_DB_PATH`, Docker test container all exist. Extend, don't rewrite. |
| C2 | **Claude Code platform** — permissions are controlled by `settings.json`, hooks by Claude Code's hook system. We work within these mechanisms. |
| C3 | **No CI yet** — Docker runner is the "CI equivalent" for now. When CI exists, the Docker runner becomes the CI test step. |
| C4 | **Valueflow A&D alignment** — quality gate tiers (T1-T4), changed-file scoping, enforcement ladder architecture, and context budget targets all come from the Valueflow A&D. DevEx implements, not redefines. |

## Success Criteria

| ID | Criterion |
|----|-----------|
| SC1 | Pre-commit hook completes in under 60 seconds for scoped changes |
| SC2 | Markdown-only commits skip tests entirely (stage-hash delta tolerance) |
| SC3 | Zero test corruption incidents — no `.git/config` modification, no DB leaks, no working directory debris |
| SC4 | All 32 BATS test files use isolation helpers |
| SC5 | `docker-test.sh` runs all 32 BATS files successfully |
| SC6 | Zero permission prompts for pre-approved operations (measurable per session) |
| SC7 | Enforcement audit tool reports 100% compliance for capabilities at their declared level |
| SC8 | No skill injection exceeds 4000 token budget |

## Non-Goals

| ID | Non-Goal |
|----|----------|
| NG1 | **CI/CD pipeline** — Docker runner is the local equivalent. CI is a future workstream. |
| NG2 | **Agent lifecycle management** — T4 (ephemeral worktree agents). Bigger design question, separate scope. |
| NG3 | **Claude Code behavioral issues** — C1 (agent permission scope), C2 (multi-principal DB security). Anthropic's domain. |
| NG4 | **agency-init rewrite** — B1 from friction points. Related (ships permissions), but its own scope. DevEx ensures the permission model is correct; agency-init ships it. |
| NG5 | **CLAUDE-THEAGENCY.md decomposition authoring** — DevEx builds the linter, not the decomposed docs. The decomposition is a Valueflow workstream deliverable. |

## Phasing

### Phase 1: Pre-commit + Isolation (the burning problem)

- Rewrite `commit-precheck`: smart scoping, non-code skip, timeout, stage-hash verification
- Extend isolation to all 32 BATS files
- Fix filesystem pollution (temp dirs, teardown guards)
- Tests for the pre-commit hook itself

### Phase 2: Docker Full Suite

- Extend `docker-test.sh` to all 32 files
- Integrate Docker as T3 mechanism
- T3 Docker fallback (if Docker unavailable, in-process with isolation, warn)

### Phase 3: Permission Model + Enforcement Tooling

- settings-template.json audit — pre-approve all safe operations
- Enforcement registry (`claude/config/enforcement.yaml`)
- Enforcement audit tool
- Context budget linter
- Permission prompt measurement

## Open Questions

**OQ1: Test mapping manifest format** — Convention-based mapping covers tools. What's the manifest format for edge cases? YAML in tool headers? Separate manifest file? Resolve in A&D.

**OQ2: settings-template.json distribution** — When DevEx updates the permission model, how does it reach existing projects? `agency update`? Manual merge? `settings-merge` tool? Resolve in Phase 3.

**OQ3: PermissionDenied hook** — Valueflow A&D §12 lists this as a V2 deliverable but has no design section. DevEx should design and own it. A hook that fires on permission denial, checks if the operation is in the safe list, and auto-approves. Requires Claude Code support — may be aspirational.
