## What Was Found and Fixed

The Phase 1A code went into the QG with 43 tests green and was the candidate for the first mdpal-app PR per captain #415. Three real issues surfaced:

1. **Sheet draft loss when `document` is nil.** All three mutation sheets (AddComment, FlagEditor, ResolveComment) declared `guard let document else { return true }` in their submit handlers. Returning `true` tells the sheet to dismiss — so a sheet attached to a stateless preview (or a future code path that constructs the view without a model) would silently discard the user's draft on submit. The intent of the guard was clearly "we can't do anything," not "throw away the work" — `return false` is the safe default. Fixed in three call sites with a comment explaining the intent.

2. **Stale `lastError` after a successful retry.** The `lastError` bus is what drives the generic alert (1A.3). The load methods (`loadSections`, `loadComments`, `loadFlags`) and `selectSection` all clear `lastError = nil` on success. The mutation methods were inconsistent: `editSection` cleared via downstream `loadSections()`, `resolveComment` via `loadComments()`, `clearFlag` via `loadFlags()` — but `addComment` only mutated the local `comments` array (no reload, no clear), and `flagSection` only cleared if `result.flagged == true`. The user-visible bug: try to add a comment, fail, see the alert; retry, succeed, alert still showing. Fixed with explicit `lastError = nil` on every mutation success path. The redundancy with downstream loads is intentional — it makes the contract obvious in the method body and survives future refactors that remove the load.

3. **Coverage symmetry.** Added the three tests called out above. They're small and hold the contract: success clears, failure sets, including for `selectSection` which previously had only a happy-path test.

The Overwrite-path "doesn't clear conflict" finding from reviewer-code looked alarming on first read but is a misread — the alert binding's setter clears `conflict` whenever the alert dismisses (any button tap), so the explicit clearing in Discard/Keep-editing buttons is belt-and-suspenders, not load-bearing. Verified by reading the `Binding(get:set:)` at lines 145–148.

The ClipboardReader-as-global-static finding is real but is Phase 1B housekeeping work, not first-PR scope. It's already on the carry-forward list in `mdpal-app-phase-1B-plan.md` and the handoff.
