## Problem Statement

Agents interact with text files through line-oriented tools — `Read` (dump N lines), `Edit` (find-and-replace a string), `Write` (overwrite the whole file). These are the `sed` and `cat` of the AI era. They work, but they have no understanding of structure. An agent editing a Python function has to read enough lines to find it, construct a unique string match, and hope the replacement doesn't break something outside its view. An agent reviewing a Markdown document re-reads the entire file to check one section.

The result is wasteful in tokens, slow, and error-prone. Agents rewrite entire documents when they should be doing scoped, section-level edits. There is no tool that lets an agent — or a human — do structured, section-oriented operations on text files.

**The specific pain in Agency workflows:** When a principal wants to comment on a PVR, A&D, or Plan, there is no good way to do it. Options today: copy-paste into chat (lose structural context, burn tokens), edit the file directly (risky, agent re-reads everything), or tell the agent in a 1B1 (only works for what the agent surfaces, not what the principal spots while reading). Review and feedback is a major friction point.

**Founding motivation:** Token economics. Agents shouldn't rewrite whole documents to change one section.

## Core Value Proposition

**Section-oriented operations on structured documents.**

Markdown Pal is a section-oriented tool that supports:

- **Structured navigation** of document structure — outline/tree views for quick traversal
- **Scoped commenting and iteration** — review state lives in the document, comments are self-contained with context
- **Diffing and AI-powered version comparison** — semantic summaries of what changed, not just mechanical diffs
- **Dual-audience workflows** — same document model, same operations for humans (GUI) and agents (CLI)
- **Token-efficient operations** — section-level reads/writes, self-contained comments, version-gated edits

**Scope:** General-purpose — any structured document that benefits from section-oriented operations. Markdown is the first format. Source code (functions, classes, methods), log files (request lifecycles, error groups), config files, and other grammar-defined formats are the aspiration.

**The bigger vision:** A paradigm shift in how agents interact with text files. From line-oriented (`sed`, `cat`, `Read`, `Edit`, `Write`) to structure-oriented (`read section`, `edit section`, `comment on section`). The engine's section model is the platform; parsers are pluggable. Markdown Pal is the first client of a broader section-oriented document engine.

## Target Users & Use Cases

**Cornerstone:** AI-augmented workflows. Every design decision serves human:agent and agent:agent collaboration. This is core tooling.

**Primary users:**
- **Human:agent pairs** — human reviews/directs in GUI, agent operates via CLI
- **Agent:agent pairs** — agents reviewing each other's work with structured, token-efficient operations

**Use cases:** Any structured document being collaboratively iterated — AIADLC artifacts (PVRs, A&Ds, Plans), articles, reports, book chapters, source code, log files. Almost always AI-augmented.

**Also works as** a standalone Markdown editor, but that's not the design driver.

## V1 Scope — The Six Capabilities

The V1 essence, stated in terms of the actual problem. Six symmetric capabilities for agents and principals:

| Capability | Agent (CLI) | Principal (App) |
|-----------|-------------|-----------------|
| Read structured | `mdpal read <section>` | Section navigation + preview |
| Edit structured | `mdpal edit <section>` | Edit in app |
| Flag for discussion | See flags, run `/discuss` | Mark sections for discussion |
| See comments | `mdpal comments` | Inline comment display |
| Add comments | `mdpal comment <section>` | Drop comments while reading |

Everything we build should serve these six capabilities. If a feature doesn't contribute to one of them, it's not V1.

### mdpal-cli V1 (Engine + CLI)

**Engine (Swift library):**
- Pluggable parser architecture with abstract structural nodes — designed from day one for multiple formats
- Markdown parser as first implementation (via `swift-markdown`)
- Section-oriented CRUD by structural address (section slugs, path-style nesting)
- Section version hashing + optimistic concurrency (compare-and-swap, no locks)
- Comment lifecycle — add, resolve, staleness detection via context field
- YAML metadata block read/write (inside HTML comment boundaries)
- `.mdpal` bundle management — create revision, update symlink, prune with metadata merge-forward

**CLI (`mdpal`):**
- Thin command-line frontend linking the engine directly
- Core commands: `sections`, `read`, `edit`, `comment`, `comments`, `resolve`, `create`, `refresh`, `history`, `diff`, `prune`
- Designed for agent invocation — structured output, no interactive prompts, exit codes

**Not in V1 for mdpal-cli:** MCP server, LSP server, template system, document homes.

### mdpal-app V1 (SwiftUI App)

**App:**
- Native macOS SwiftUI app (iPadOS possible future)
- Section navigation sidebar — outline/tree view of document structure
- Inline comment display with interactive affordances (resolve, respond)
- Comment threading and resolution workflow
- Diff view for comparing revisions within a bundle
- Version history browser

**Bundle layer (.mdpal):**
- Bundle packaging and OS presentation (package UTType, Finder integration)
- Revision management — when to create new revisions, symlink updates
- `DocumentGroup`-based with `ReferenceFileDocument` and `FileWrapper`

**Not in V1 for mdpal-app:** MCP integration, LSP adapter, template browser, document home navigation, iOS/iPadOS port.

## Platform & Language

**Language:** Swift. Shared language, types, and toolchain between engine and app. The engine is a Swift package that both CLI and app link directly.

**Platforms:**
- **macOS:** Primary. First-class support.
- **Linux:** Second. Swift on Linux is production-ready (`swift-markdown`, `swift-argument-parser`, `Yams` all work cross-platform). Engine + CLI will target Linux.
- **Windows:** Deferred. Active blocker (`swift-markdown` issue #244). Not our market unless enterprise.

## Architecture — Engine as Library

**Phase 1:** Engine library with pluggable parsers and section-oriented API. No protocol layer.

**Phase 2:** CLI + SwiftUI app in parallel, both linking the library directly. No serialization, no handshake, no protocol overhead.

**Protocol adapters (future):** MCP and LSP come when remote clients need them. The API shape crystallizes through building the CLI and app first.

**Why no LSP in Phase 1:** LSP is line-oriented (line/character positions). Our engine is section-oriented (structural addresses). LSP assumes the editor owns the document and the server adds intelligence. In our model, the engine owns the document and clients operate on it. We're closer to a database with a query API than an LSP server. The engine's native API — section-oriented, AST-based — is the foundation. LSP and MCP are protocol adapters on top, not the core. (Full analysis in session 2 transcript.)

**Engine API — two layers:**
1. **Core:** Section-oriented operations on any file (read, edit, comment, diff) — format-agnostic via pluggable parsers
2. **Bundle:** Revision management operations (.mdpal packaging) — Markdown Pal-specific

The core layer is the platform. The bundle layer is product-specific. Both live in the engine library.

## Agent Interface Priority

**CLI is the agent interface.** Any agent that can run shell commands gets full access. Claude Code invokes `mdpal read authentication` as a Bash tool call — no integration layer needed.

**MCP:** Deferred to "if we ever actually need it." The CLI already gives agents everything MCP would, minus persistent in-memory AST and push notifications. Those capabilities may never be worth the complexity.

**LSP:** An adapter for traditional editors (VS Code, Neovim). Not an agent interface. mdpal-app's domain if and when editor integration is desired.

## Bundle Format (.mdpal)

**Confirmed.** The `.mdpal` bundle is an **optional document packaging layer**, not an engine requirement. The engine operates on raw files. The bundle adds revision management and review state on top.

- macOS (and possibly iOS) — Apple ecosystem, symlinks, FileWrapper, package UTTypes all natively supported
- Git tracks everything normally — these are small text files (~20-30K), no bloat concern
- Source code and other non-Markdown formats use the engine's section operations directly on files in place, without bundles

**Bundle is an app concern.** The engine provides bundle management operations (create revision, update symlink, prune). The app decides when and how to use them.

## Coordination Model

**Parallel with contract.** Two agents, one product:

- **mdpal-cli** builds the engine library + CLI
- **mdpal-app** builds the SwiftUI app + bundle layer

API contract defined in A&D. Both agents work independently against the contract. Dispatches handle exceptions when the app discovers an engine need: "Hey, I want this. I did it on this branch. Grab it and run with it!"

## Relationship to The Agency

**mdpal is infrastructure for Agency workflows.** It solves the review and feedback friction between principals and agents. When a principal wants to comment on a PVR, A&D, or Plan — or when an agent wants to review another agent's document — mdpal provides structured, token-efficient, section-oriented operations with a preserved record.

## Competitive Landscape

**Nothing fills the gap.**

| Category | Examples | Gap |
|----------|----------|-----|
| Markdown editors | Typora, Obsidian, iA Writer, MarkEdit | Authoring, not structured review. No section-oriented operations, no comment model, no agent interface |
| Code review tools | GitHub PRs, Reviewable, Gerrit | Line-oriented diffs on committed code. Not for iterating on documents |
| Collaborative docs | Google Docs, Notion | Closest UX model for comments, but not Markdown, not local-first, no agent interface, no CLI |
| LSP Markdown tools | Marksman | Navigation and diagnostics only. No review/comment model |

No existing tool combines structured document operations with a comment/review model that both humans and agents can use.

## Non-Goals (V1)

- Windows support
- MCP server
- LSP server / editor integration
- Template system and template inheritance
- Document homes
- Cross-artifact comment views
- Push notifications on section changes
- Prompt packaging / prompt recipes
- iOS / iPadOS port
- Enterprise features

## Success Criteria

1. An agent can read a specific section of a Markdown file without reading the whole file
2. An agent can edit a specific section with optimistic concurrency (version hash)
3. A principal can drop comments on sections in the GUI and the agent sees them via CLI
4. An agent can add comments via CLI and the principal sees them in the GUI
5. Comments are self-contained (carry their own context) and support staleness detection
6. The principal can flag sections for 1B1 discussion and the agent can see the flags
7. The engine handles Markdown via a pluggable parser interface that is designed (not bolted on) for future formats
