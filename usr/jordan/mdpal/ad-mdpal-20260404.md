## 1. System Overview

Markdown Pal is a section-oriented tool for structured document operations. The system has three independent components communicating via CLI commands and ISCP dispatches:

```
┌──────────────────────────────────────────────────┐
│              mdpal Engine + CLI                   │
│  ┌───────────────────┐  ┌────────────────────┐   │
│  │    Core Layer      │  │   Bundle Layer     │   │
│  │  (format-agnostic) │  │  (mdpal-specific)  │   │
│  └────────┬──────────┘  └────────┬───────────┘   │
│           │                      │                │
│  ┌────────┴──────────┐           │                │
│  │  Parser Protocol   │           │                │
│  │  ┌──────────────┐ │           │                │
│  │  │MarkdownParser│ │           │                │
│  │  └──────────────┘ │           │                │
│  │  ┌──────────────┐ │           │                │
│  │  │ (future)     │ │           │                │
│  │  └──────────────┘ │           │                │
│  └───────────────────┘           │                │
│           │                      │                │
│     ┌─────┴─────┐                │                │
│     │    CLI    │                │                │
│     │  (mdpal)  │                │                │
│     └───────────┘                │                │
└──────────┬───────────────────────┘                │
           │                                         │
    CLI commands (JSON)                              │
    ISCP dispatches                                  │
           │                                         │
┌──────────┴───────────────────────────────────────┐
│              Markdown Pal App                     │
│  ┌───────────────┐  ┌──────────────────────────┐ │
│  │  FileWrapper   │  │   SwiftUI Document Model │ │
│  │  (I/O owner)   │  │   (app-side types)      │ │
│  └───────────────┘  └──────────────────────────┘ │
│  ┌───────────────────────────────────────────┐   │
│  │  CLI Integration Layer (Process wrapper)   │   │
│  └───────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
```

**Engine + CLI** — Swift package providing section-oriented operations and a command-line interface. Two engine layers:
- **Core:** Format-agnostic section operations (read, edit, comment, flag, diff) via pluggable parsers
- **Bundle:** `.mdpal` bundle management (revisions, pruning, metadata merge-forward)

The CLI is a thin command-line frontend. Stateless: parse, operate, exit. Links the engine directly. **The CLI's JSON command interface is the public contract** — all external consumers (app, agents, future tools) interact through CLI commands.

**App** — Independent native macOS SwiftUI package. Stateful: loads document into memory via FileWrapper, holds its own document model. **Does not link the engine library.** Invokes `mdpal` CLI commands via `Process` and parses JSON responses into app-side Swift types.

**Communication layer:** CLI commands (synchronous operations) + ISCP dispatches (asynchronous workflows like review requests, comment threads, change notifications). No direct library linking between app and engine. No filesystem notifications. No engine callbacks.

**Principle:** The tooling works equally well for principals (humans via the app) and agents (via CLI commands). The app experience (UX/HX) and the agent experience (AX) evolve independently, interacting only through CLI commands and ISCP dispatches. This also makes it easy to plug in support for other structured document types — new parsers register with the engine, and both app and agents gain access through the same CLI interface.

Protocol adapters (MCP, LSP) are future work when remote clients need them.

## 2. Pluggable Parser Protocol

The engine's core layer operates on abstract structural nodes, not Markdown-specific types. Parsers bridge between a file format and the engine's structural model.

### 2.1 What Parsers Implement

```swift
/// A parser converts raw text content into a structural tree
/// that the engine can operate on.
protocol DocumentParser {

    /// The file extensions this parser handles (e.g., [".md", ".markdown"])
    static var supportedExtensions: [String] { get }

    /// Parse raw text content into a structural tree.
    /// Returns the root node of the section tree.
    func parse(_ content: String) throws -> SectionTree

    /// Serialize a (possibly modified) section tree back to text.
    /// Must round-trip faithfully: parse(serialize(tree)) ≈ tree
    /// for content not modified by the engine.
    func serialize(_ tree: SectionTree) throws -> String

    /// Compute the slug for a section heading.
    /// Each format may have its own slug rules.
    func slug(for heading: String) -> String

    /// Detect metadata block boundaries in the raw text.
    /// Returns nil if no metadata block exists.
    func findMetadataBlock(in content: String) -> MetadataRange?

    /// Insert or replace a metadata block in the raw text.
    func writeMetadataBlock(_ yaml: String, into content: String) -> String
}
```

### 2.2 What the Engine Provides (Parser-Independent)

The engine handles everything that is not format-specific:

- **Comment storage and lifecycle** — add, resolve, staleness detection
- **Flag storage and lifecycle** — set, clear, list
- **Version hashing** — SHA-256 of section content, truncated to 6 hex chars
- **Metadata serialization** — YAML read/write via Yams
- **Optimistic concurrency** — compare-and-swap on section version hashes
- **Section addressing** — slug computation delegates to parser, path assembly is engine logic
- **Bundle management** — revision creation, symlink updates, pruning, metadata merge-forward

### 2.3 Structural Model

```swift
/// The parsed representation of a document's structure.
/// SectionTree and SectionNode are engine-internal types, not
/// exposed in the public API. Consumers use Section and SectionInfo.
struct SectionTree {
    let root: SectionNode

    /// The original source text. Retained for byte-range slicing
    /// during serialization (see implementation note in 2.5).
    let originalSource: String
}

/// A node in the section tree. Represents a heading and its content.
struct SectionNode {
    /// The heading text (raw, before slug computation)
    let heading: String

    /// The heading level (1-6 for Markdown; parser-defined for other formats)
    let level: Int

    /// The raw content of this section: body text only, EXCLUDING
    /// the heading line and EXCLUDING child sections. When editSection()
    /// replaces content, it replaces this body — the heading is preserved,
    /// and child sections are re-parsed from the new content.
    let content: String

    /// The source range in the original document (UTF-8 byte offsets).
    /// Used for faithful serialization via byte-range slicing of the
    /// original source — see implementation note in section 2.5.
    let sourceRange: Range<String.Index>

    /// Child sections (subsections under this heading)
    var children: [SectionNode]
}
```

The root node represents the document itself (level 0). Top-level sections are its children. Content before the first heading becomes the root node's body content. Documents with no headings have all content in the root node with `children: []`. The root node's `heading` is an empty string.

## 3. Engine API Contract

The engine's Swift API. This is the **internal API** used by the CLI to implement commands. The **public contract** between components is the CLI command interface (§9) and ISCP dispatch format — not these Swift types directly.

The app does not link this library. It invokes CLI commands and parses JSON output into its own app-side types. The Swift types below are documented here because they define the engine's capabilities and the semantics that the CLI commands expose.

### 3.1 Document — The Central Type

```swift
/// A parsed, queryable representation of a structured document.
/// Supports two initialization modes:
/// - From file path (CLI mode): engine owns file I/O
/// - From raw data (library mode): caller owns file I/O
class Document {

    /// The section tree (parsed structure)
    let sections: SectionTree

    /// The metadata (comments, flags, document info)
    var metadata: DocumentMetadata

    /// The parser used for this document
    let parser: DocumentParser

    // -- Initialization --

    /// CLI mode: engine reads the file, owns I/O.
    /// Throws if file not found, unreadable, or unparseable.
    /// Synchronous — local file I/O does not benefit from async.
    convenience init(contentsOfFile path: String) throws

    /// Library mode: caller provides raw content, engine parses.
    /// No file I/O. Pure transform.
    init(content: String, parser: DocumentParser) throws

    /// Library mode with explicit format detection.
    /// Resolves parser via ParserRegistry.parser(for: fileExtension).
    /// The registry is a static list of known parser types; each parser
    /// declares its supported extensions via DocumentParser.supportedExtensions.
    init(content: String, fileExtension: String) throws

    // -- Section Operations (synchronous — operates on in-memory tree) --

    /// List all sections with their slugs, levels, and version hashes.
    func listSections() -> [SectionInfo]

    /// Read a section by slug. Path-style addressing for nested sections.
    /// Throws .sectionNotFound if slug doesn't match.
    func readSection(_ slug: String) throws -> Section

    /// Edit a section's body content with optimistic concurrency.
    /// newContent replaces the section body (not the heading, not child sections).
    /// The engine re-parses newContent to discover any new child sections within it.
    /// The versionHash must match the current section hash.
    /// Throws .versionConflict(currentSection:) if hash doesn't match.
    func editSection(_ slug: String, newContent: String,
                     versionHash: String) throws -> Section

    /// Refresh stale comments on a section: updates each comment's
    /// versionHash to the section's current hash. Does NOT update
    /// comment context — the original context is preserved so the
    /// reader can see what the comment was written against.
    /// Returns the list of refreshed comments.
    func refreshSection(_ slug: String) throws -> [Comment]

    // -- Comment Operations --

    /// Add a comment anchored to a section.
    func addComment(_ comment: NewComment) throws -> Comment

    /// List all comments, optionally filtered.
    func listComments(filter: CommentFilter?) -> [Comment]

    /// Resolve a comment with a response.
    func resolveComment(id: String, response: String,
                        resolvedBy: String) throws -> Comment

    // -- Flag Operations --

    /// Flag a section for discussion.
    func flagSection(_ slug: String, note: String?,
                     author: String) throws -> Flag

    /// Clear a flag on a section.
    func clearFlag(_ slug: String) throws

    /// List all flagged sections.
    func listFlags() -> [Flag]

    // -- Diff --

    /// Diff a single section against previous content.
    func diffSection(_ slug: String,
                     againstContent: String) -> SectionDiff

    /// Diff this document against another document. Returns per-section
    /// diffs plus added/removed sections. Used internally by
    /// DocumentBundle.diffRevisions() and available for testing.
    func diff(against other: Document) -> DocumentDiff

    // -- Serialization --

    /// Serialize the document back to text (with updated metadata).
    /// Returns the full document content ready to write to disk.
    func serialize() throws -> String
}
```

### 3.2 Data Types


### 3.3 Error Model

```swift
enum EngineError: Error {
    /// Section not found for the given slug.
    case sectionNotFound(slug: String, availableSlugs: [String])

    /// Version conflict — section has been modified since read.
    /// Includes current content and hash so caller can retry.
    case versionConflict(
        slug: String,

    /// Comment not found.
    case commentNotFound(id: String)

    /// Comment already resolved.
    case commentAlreadyResolved(id: String)

    /// Section not flagged (when trying to clear a flag).
    case sectionNotFlagged(slug: String)

    /// Parse error — document content could not be parsed.

    /// No parser available for the given file extension.
    case unsupportedFormat(extension: String)

    /// File I/O error (CLI mode only).
    case fileError(path: String, underlying: Error)

    /// Metadata block is malformed (YAML parse failure).
    case metadataError(description: String)
```

### 3.4 Engine State: Stateless vs Stateful

The engine supports two usage patterns through `Document` initialization:

**CLI mode (stateless):** Parse, operate, exit. Each CLI invocation creates a `Document` from a file path, performs one operation, serializes the result back to disk, and exits. No in-memory state persists between invocations.

```swift
// CLI: read a section
let doc = try Document(contentsOfFile: "design.mdpal/latest.md")
let section = try doc.readSection("authentication")
print(section.content)
// Process exits. No state retained.
```

**Library mode (for testing and embedding):** The `content:parser:` initializer exists for unit/integration testing and for any future consumer that embeds the engine directly. The app does not use this — it invokes CLI commands instead.

```swift
// Library mode: caller provides content, engine parses
let doc = try Document(content: markdownString, parser: MarkdownParser())
let section = try doc.readSection("authentication")
// ... operate on the in-memory document ...
let output = try doc.serialize()
```

**Key design constraint:** The `Document` class is **not** thread-safe. Callers must serialize access (e.g., via `@MainActor` or a serial dispatch queue). The CLI doesn't have this concern since it's single-operation-per-invocation.

### 3.5 App-Engine Communication Model (r003)

The app and engine are independent packages. They communicate through two mechanisms:

**CLI commands (synchronous):** The app invokes `mdpal` commands via `Process`, passing arguments and receiving JSON on stdout. Used for all document operations: list sections, read, edit, comment, flag, diff, revision create.

**ISCP dispatches (asynchronous):** Used for workflow-level communication that doesn't map to a single CLI call. Examples:
- Agent prepares a PVR and sends a review-request dispatch → appears in app's tray → principal reviews and comments → feedback flows back as ISCP dispatches
- Engine detects a comment thread needs attention → dispatch to app/principal
- App-initiated review feedback → dispatch back to the agent

**No filesystem notifications.** The app does not watch the bundle directory for changes. The engine does not emit callbacks or notifications. When the app needs fresh data, it re-invokes CLI commands. ISCP dispatches handle the "something changed, you should look" notification.

**No direct library linking.** The app parses CLI JSON output into its own Swift types. The engine's internal `Document`, `Comment`, `Flag` types are not shared across the boundary. This means the app and engine can evolve independently — the CLI JSON format is the stability contract.

**Rationale:** This architecture serves both human and agent experiences equally. Agents already interact via CLI commands. ISCP dispatches enable the review workflow Jordan envisioned: agent dispatches review to principal, appears in mdpal app tray, principal reviews and comments, feedback flows back as ISCP dispatches. The same dispatch mechanism works for agent-to-agent coordination. It also makes plugging in new document format parsers seamless — register with the engine, both app and agents gain access through the same CLI.

### 3.6 File I/O Ownership (r004)

Two distinct I/O paths, fully separated by component boundary:

| Component | Who owns file I/O | How content is accessed | When content is written |
|-----------|-------------------|------------------------|------------------------|
| Engine/CLI | Engine | `Document(contentsOfFile:)` | Engine writes after operation; Bundle creates revisions |
| App | App (via FileWrapper) | Reads file, invokes CLI commands | App writes working copy via FileWrapper; explicit save shells out to `mdpal revision create` |

The engine provides convenience methods for CLI-mode file writes:

```swift
extension Document {
    /// Write the serialized document back to a file.
    /// CLI mode only. Throws if called on a document
    /// created from raw content (no file path).
    func write() throws

    /// Write to a specific path (for creating new revisions).
    func write(to path: String) throws
}
```

**The app never calls any engine API directly.** It invokes CLI commands via `Process`. For document operations, the app calls CLI commands like `mdpal read`, `mdpal edit`, `mdpal comment`. For explicit save (⌘S), the app shells out to `mdpal revision create --stdin`, piping the serialized content. Auto-save writes the working copy through FileWrapper — no CLI call, no revision.

**CLI write flow for bundles:** When the CLI operates on a file inside a `.mdpal` bundle, it uses `DocumentBundle` — not `Document.write()`. The flow is: `DocumentBundle(at:)` → `currentDocument()` → operate on `Document` → `bundle.createRevision(content: doc.serialize())`. `Document.write()` is for standalone `.md` files only (not inside bundles). The CLI detects whether a path is inside a bundle by checking for the `.mdpal/config.yaml` directory structure.

**Relationship between Document and DocumentBundle:** `Document` operates on a single file's content (parsing, section ops, comments, flags). `DocumentBundle` operates on the bundle directory (revisions, pruning, symlinks). `DocumentBundle.currentDocument()` loads the latest revision as a `Document`. `DocumentBundle.diffRevisions()` loads two revisions as Documents and calls `Document.diff(against:)` internally.

## 4. Section Addressing

### 4.1 Slug Computation

Slugs are computed by the parser (format-specific). For Markdown:

1. Strip inline Markdown formatting: `**bold**` → `bold`, `` `code` `` → `code`, `*italic*` → `italic`, `[link](url)` → `link`
2. Convert to lowercase
3. Replace all non-alphanumeric characters (except hyphens) with hyphens
4. Collapse consecutive hyphens to a single hyphen
5. Trim leading and trailing hyphens

Examples:

| Heading | Slug |
|---------|------|
| `## Authentication` | `authentication` |
| `## CLI Design (\`mdpal\`)` | `cli-design-mdpal` |
| `## Open Questions` | `open-questions` |
| `### OAuth` (under `## Authentication`) | `authentication/oauth` |
| `## V1 Scope — The Five Capabilities` | `v1-scope-the-five-capabilities` |

### 4.2 Path-Style Nesting

Nested sections use `/` as the path separator. The full slug path is assembled by the engine (not the parser):

```
authentication              → ## Authentication
authentication/oauth        → ### OAuth (under ## Authentication)
authentication/oauth/scopes → #### Scopes (under ### OAuth)
```

The engine resolves paths by walking the section tree. `readSection("authentication/oauth")` finds the `## Authentication` node, then finds `### OAuth` among its children.

### 4.3 Duplicate Heading Disambiguation

When two headings at the same level produce the same slug, the engine appends a positional suffix:

```markdown
## Examples        → examples
## Details
### Examples       → details/examples
## Examples        → examples-2
```

Rules:
- First occurrence: bare slug (`examples`)
- Subsequent occurrences: append `-N` where N starts at 2 (`examples-2`, `examples-3`)
- Disambiguation is positional (document order), not content-based
- Nested duplicates are disambiguated within their parent scope

The `listSections()` output always includes the disambiguated slug, so callers never need to compute disambiguation themselves.

**Stability:** Disambiguation is **positional** — recomputed on every parse based on document order. If the first `## Examples` heading is deleted, the former `examples-2` becomes `examples`. This means slug-anchored comments may become stale when headings are added or removed. This is acceptable: the staleness detection mechanism (version hash comparison) already handles this case. Comments anchored to a renamed/reindexed slug will be flagged as stale.

### 4.4 Sub-Section Granularity

**V1 stops at section granularity for the engine API.** The engine's unit of operation is the section (heading + content until next heading of equal or higher level).

Sub-section addressing (e.g., `paragraph:2`, `paragraph:contains("Redis")`) from the seed doc is **not** part of the engine API. If the CLI wants to support sub-section filtering, it reads the full section and filters locally — this is a CLI convenience, not an engine contract.

**Rationale:** Section granularity covers the PVR's five capabilities. Sub-section addressing adds API complexity without a clear V1 use case. It can be added to the engine API in a future version if demand emerges.

## 5. Comment and Flag Data Model

### 5.1 Metadata Block Format

Comments, flags, and document metadata are stored in a YAML block at the end of the file, inside HTML comment boundaries with a fenced code block.

**Invariant:** The metadata block MUST be preceded by a blank line. Per the CommonMark spec, `swift-markdown` only recognizes `<!-- ... -->` as an `HTMLBlock` when preceded by a blank line. Without the blank line, the HTML comment may be absorbed into a preceding paragraph as raw inline HTML, which would break serialization. The `writeMetadataBlock()` method enforces this invariant.

````
end:markdown-pal-meta -->
````

### 5.2 YAML Round-Tripping

The metadata block lives inside HTML comment boundaries. `swift-markdown` parses `<!-- ... -->` as `HTMLBlock` nodes — opaque text, not structured data. The engine's Markdown parser handles this as follows:

1. **Detection:** Scan for `<!-- begin:markdown-pal-meta` as a string match in the raw content. This is done by the parser's `findMetadataBlock()` method, not by walking the AST.
2. **Extraction:** Extract the content between the boundary markers. Strip the fenced code block markers (` ```yaml ` and ` ``` `). Hand the raw YAML string to Yams.
3. **Parsing:** Yams parses the YAML into the engine's `DocumentMetadata` type.
4. **Modification:** The engine modifies the in-memory `DocumentMetadata` (add comment, resolve comment, set flag, etc.).
5. **Serialization:** The engine serializes `DocumentMetadata` back to YAML via Yams, wraps it in the fenced code block + HTML comment boundaries, and replaces the metadata block in the document content.

**Round-trip fidelity:** The engine does not attempt to preserve YAML formatting (key order, quoting style, whitespace). It serializes cleanly each time. This means the metadata block may reformat on write, but the content is semantically identical. This is acceptable because the metadata block is machine-managed, not hand-edited.

**Edge case — nested backticks:** If any YAML value contains triple backticks, the fenced code block would be prematurely terminated. The `writeMetadataBlock()` method uses a longer fence (e.g., `` `````yaml ``) to reduce collision risk. Unlikely in practice but must be handled.

**Edge case — no metadata block:** If a file has no metadata block, the engine creates one on first write (append to end of file, preceded by a blank line per the invariant above).

**Edge case — malformed metadata:** If the YAML is unparseable, the engine throws `EngineError.metadataError`. It does not silently discard or overwrite malformed metadata.

### 5.3 Comment ID Generation

Comment IDs are generated by the engine using a compact format: `c` + zero-padded sequential number within the document (`c001`, `c002`, ..., `c999`, `c1000`).

The engine determines the next ID by scanning existing comments (both unresolved and resolved) and incrementing from the highest.

### 5.4 Staleness Detection

A comment is **stale** when the section it refers to has changed since the comment was written. Detection:

1. The comment stores `versionHash` — the section's hash when the comment was created.
2. Compare against the section's current hash.
3. If different, the comment is stale.

The engine provides:

```swift
extension Comment {
    /// Whether this comment's section has changed since the comment was created.
    func isStale(in document: Document) -> Bool
}
```

Staleness is informational — stale comments are not automatically removed. The CLI reports staleness; the app displays a visual indicator. Resolution is a human/agent decision.

### 5.5 Context Capture

When a comment is created without an explicit `context` field, the engine automatically captures the current section content as context. This ensures comments are self-contained — reading a comment gives you the text it was written against without a separate section read.

The `context` field is always populated. It is never empty.

### 5.6 Flag Storage

Flags are stored in the metadata block under `flags:` as a flat list. Each flag has: `section` (slug), `note` (optional), `author`, `timestamp`.

Flags are section-level — one flag per section. Flagging an already-flagged section replaces the existing flag (updates author, note, timestamp).

## 6. Bundle Operations

### 6.1 Bundle Structure

```
{Name}.mdpal/
├── latest.md → V0001.0003.20260404T1200Z.md    (symlink — for CLI/agents)
├── V0001.0003.20260404T1200Z.md                 (current revision)
├── V0001.0002.20260403T1500Z.md                 (previous)
├── V0001.0001.20260403T0900Z.md                 (initial)
└── .mdpal/
    ├── config.yaml
    └── latest                                    (pointer file — for app/FileWrapper)
```

**Dual latest mechanism:** Two ways to find the current revision:
- **Symlink** (`latest.md`): For CLI and agents. Standard Unix symlink. Fast, atomic updates.
- **Pointer file** (`.mdpal/latest`): For the app/FileWrapper. Contains the filename of the current revision as plain text (e.g., `V0001.0003.20260404T1200Z.md`). FileWrapper dereferences symlinks transparently, making the symlink unreliable for detecting the current revision from the app side.

The engine maintains both atomically: when creating a revision, it updates the symlink and writes the pointer file in sequence. Both always point to the same revision.

**Note:** `DocumentBundle` is a CLI/engine-only type. The app never calls it directly — it invokes CLI commands like `mdpal revision create` and `mdpal history` instead.

### 6.2 Bundle Configuration

`.mdpal/config.yaml`:

```yaml
name: MarkdownPalDesign
prune:
  keep: 20
  auto: false
```

V1 fields: `name`, `prune.keep`, `prune.auto`. Template and document home fields from the seed doc are deferred (non-goals in PVR).

### 6.3 Version ID Format

`V{NNNN}.{NNNN}.{YYYYMMDD}T{HHMM}Z`

- **Version** (4 digits): Bumped explicitly. Human decision.
- **Revision** (4 digits): Auto-incremented on each save.
- **Timestamp**: UTC, compact ISO 8601.

Examples: `V0001.0000.20260404T0900Z`, `V0001.0003.20260404T1430Z`, `V0002.0000.20260405T1000Z`

### 6.4 Bundle API

```swift
/// Bundle management operations.
/// Named DocumentBundle to avoid shadowing Foundation.Bundle.
/// These are engine-provided algorithms that operate on the bundle directory.
struct DocumentBundle {

    let path: String          // Path to the .mdpal directory
    let config: BundleConfig  // Parsed config.yaml

    // -- Initialization --

    /// Open an existing bundle.
    init(at path: String) throws

    /// Create a new bundle with an initial revision.
    static func create(
        name: String,
        initialContent: String,
        at directory: String
    ) throws -> DocumentBundle

    // -- Revision Operations --

    /// Get the current (latest) document.
    func currentDocument() throws -> Document

    /// Create a new revision from modified content.
    /// Increments revision number, writes new file, updates symlink.
    func createRevision(content: String) throws -> RevisionInfo

    /// Bump the version number (resets revision to 0).
    func bumpVersion() throws -> RevisionInfo

    /// List all revisions in the bundle.
    func listRevisions() throws -> [RevisionInfo]

    /// Diff two revisions.
    func diffRevisions(_ rev1: String, _ rev2: String) throws -> DocumentDiff

    // -- Pruning --

    /// Prune old revisions, keeping the most recent N.
    /// Merges forward any resolved comment history from pruned revisions.
    func prune(keep: Int) throws -> PruneResult

    // -- Configuration --

    /// Update bundle configuration.
    func updateConfig(_ config: BundleConfig) throws
}

struct RevisionInfo {
    let versionId: String     // e.g., "V0001.0003.20260404T1200Z"
    let version: Int
    let revision: Int
    let timestamp: Date
    let filePath: String
}

struct BundleConfig: Codable {
    var name: String
    var prune: PruneConfig

    struct PruneConfig: Codable {
        var keep: Int
        var auto: Bool
    }
}

struct PruneResult {
    let prunedRevisions: [String]     // Version IDs of deleted revisions
    let mergedComments: Int           // Number of comments merged forward
    let remainingRevisions: Int       // How many revisions remain
}

struct DocumentDiff {
    let rev1: String
    let rev2: String
    let sectionDiffs: [SectionDiff]   // Per-section diffs
    let addedSections: [String]       // Slugs of sections added in rev2
    let removedSections: [String]     // Slugs of sections removed in rev2
}
```

### 6.5 Pruning Algorithm

When `prune(keep: N)` is called:

1. **List** all revisions sorted by version ID (lexicographic — the format is designed for this).
2. **Identify** revisions to prune: all except the N most recent.
3. **Merge forward** resolved comment history:
   a. For each pruned revision, read its metadata block.
   b. For each resolved comment in a pruned revision, check if it exists in the current (latest) revision's resolved list (matched by comment ID).
   c. If not, append it to the current revision's resolved list.
   d. Write the updated metadata back to the latest revision file.
4. **Gate:** Verify the symlink still points to the same revision as when pruning started. If another writer created a new revision during pruning, abort and report `bundleConflict`. This uses the same revision-gating mechanism as section 7.2.
5. **Delete** the pruned revision files.
6. **Verify** the symlink still points to a valid file (it should — we never prune the latest).

**Invariant:** After pruning, the latest revision's metadata contains the complete resolved comment history. No comment history is lost.

**Auto-prune:** If `config.prune.auto` is true, the engine checks the revision count after `createRevision()` and prunes automatically if it exceeds `config.prune.keep`.

### 6.6 Latest Pointer Management

Both `latest.md` (symlink) and `.mdpal/latest` (pointer file) always refer to the most recent revision file.

- `createRevision()` writes the new file, then atomically updates the symlink (create new symlink at temp path, rename over existing), then writes the pointer file.
- `bumpVersion()` is a special case of `createRevision()` that resets the revision counter and increments the version counter.
- On Linux: symlinks work identically. Pointer file is cross-platform.
- On Windows: pointer file is the primary mechanism (symlinks require elevated privileges). Deferred — not a V1 target.

**Update order:** Symlink first, then pointer file. If the process crashes between the two writes, the symlink is authoritative. On next engine operation, the engine detects the mismatch and repairs the pointer file.

### 6.7 Revision Triggers (App vs CLI)

| Trigger | Who decides | Mechanism |
|---------|-------------|-----------|
| CLI write operation | Engine (automatic) | After edit/comment/resolve/flag, engine writes a new revision via DocumentBundle |
| App explicit save (⌘S) | App | App shells out to `mdpal revision create <bundle> --stdin`, piping serialized content. CLI creates revision, updates symlink + pointer file. Synchronous — app gets immediate success/failure. |
| App auto-save | App (SwiftUI lifecycle) | App writes working copy via FileWrapper — NOT a new revision. No CLI call. No bundle operation. |

**Key distinction:** App auto-save (SwiftUI lifecycle) ≠ bundle revision. Two separate paths:
- **Auto-save:** FileWrapper writes working copy. Preserves work-in-progress. Fast, no overhead.
- **Explicit save:** Shells out to CLI. Creates a versioned, addressable revision. Slightly slower but provides the version history that makes the bundle model valuable.

## 7. Concurrent Write Strategy (r002)

### 7.1 Threat Model

| Scenario | Risk | V1 scope |
|----------|------|----------|
| CLI + CLI (sequential) | None — CLI is stateless, serial | N/A |
| CLI + CLI (parallel) | Low — unlikely in practice | V1 |
| App + CLI (simultaneous) | **Primary risk** — app holds working copy in memory (app-side types), CLI modifies file on disk | V1 design, Phase 2 implementation |
| App + App (same bundle) | Low — single-user product | Out of scope |

### 7.2 V1 Strategy: File-Level Optimistic Concurrency

The CLI already uses **section-level** optimistic concurrency (version hash compare-and-swap). For file-level concurrent access, we add **bundle-level revision gating:**

1. Before writing, the CLI checks the current `latest.md` symlink target.
2. The CLI reads from a specific revision (captured at operation start).
3. On write, the CLI verifies the symlink still points to the same revision.
4. If the symlink has changed (another writer created a new revision), the CLI reports a conflict and returns the current content.

**Known V1 limitation:** There is a TOCTOU (time-of-check-to-time-of-use) race between the verify and the actual file write. Two CLI invocations racing could both pass the check. This is acknowledged as acceptable for V1 — the window is extremely small, and the advisory lock in Phase 2 eliminates it. Do not attempt to solve this with complex locking in V1.

```swift
enum EngineError {
    // ... existing cases ...

    /// Bundle has been modified by another writer since this operation began.
    case bundleConflict(
        expectedRevision: String,
        currentRevision: String,
        currentDocument: Document
    )
}
```

### 7.3 Phase 2: Advisory Lock File

When the app ships and both CLI and app operate on the same bundle, we add an advisory lock:

```
{Name}.mdpal/
└── .mdpal/
    ├── config.yaml
    └── lock            ← advisory lock file
```

The lock file contains: `{"holder": "app", "pid": 12345, "timestamp": "..."}`. The CLI checks the lock before writing. If locked by the app, the CLI returns an error with a message ("bundle is open in Markdown Pal — close the app or use the app to make changes").

This is an **advisory** lock — not enforced by the OS. It handles the common case (user has app open, agent runs CLI) without requiring OS-level file locking that doesn't work across all platforms.

### 7.4 API Design for Future-Proofing

The V1 API does not need lock parameters. The `bundleConflict` error case is sufficient. Phase 2 adds lock checking internally to the engine — no API change required. The error type already covers the concurrent access case.

## 8. Version Bump Enforcement (r008)

### 8.1 Decision: Convention, Not Enforcement

Version bumps are a human decision. The engine does not enforce who can bump. The CLI provides `mdpal version bump <bundle>`, and both humans and agents can call it.

**Rationale:**
- Enforcement requires caller identity, which the engine doesn't and shouldn't track.
- The PVR says "agents do not bump versions" — this is a workflow convention, not an engine constraint.
- In Agency workflows, the agent knows it shouldn't bump because the skill/instruction says so.
- Adding `--author` flags or identity verification would add complexity for a convention that's better enforced socially.

### 8.2 Version Bump Operation

`bumpVersion()` on a Bundle:
1. Increment the version number.
2. Reset the revision to 0.
3. Copy the current latest content to the new version file.
4. Update the symlink.
5. Write the new version ID to the metadata block.

## 9. CLI Command Specification

### 9.1 General Conventions

- **Output:** JSON on stdout by default. Human-readable text when `--format text` is passed. No `isatty()` auto-detection — agents run in ptys that would incorrectly trigger text mode.
- **Errors:** stderr, structured JSON format: `{"error": "type", "message": "...", "details": {...}}`. Errors beyond the dedicated exit codes use exit 1; callers should parse the stderr JSON `error` field for programmatic error discrimination.
- **Exit codes:** 0 = success, 1 = general error, 2 = version conflict, 3 = section not found, 4 = bundle conflict.
- **Bundle argument:** All commands that operate on a bundle accept either the `.mdpal` directory path or a relative path. The CLI resolves `latest.md` automatically.

### 9.2 Command Reference

```
mdpal create <name> [--dir <path>]
```
Create a new `.mdpal` bundle with an initial empty revision. Sets up bundle structure, config.yaml, initial revision file, and latest.md symlink.

```
mdpal sections <bundle>
```
List all sections with slugs, heading levels, and version hashes.

```
mdpal read <slug> <bundle>
```
Read a section's content. Returns the section content and version hash. Use path-style slugs for nested sections: `mdpal read authentication/oauth design.mdpal`.

```
mdpal edit <slug> --version <hash> <bundle> [--content <text> | --stdin]
```
Edit a section's content. Requires the version hash for optimistic concurrency. Content via `--content` flag or stdin (reads until EOF). Content replaces the section body only — heading and child sections are preserved. On version conflict, exits with code 2 and returns the current content + new hash on stdout.

```
mdpal comment <slug> <bundle> --type <type> --author <author> --text <text> [--context <text>] [--priority <low|normal|high>] [--tags <comma-separated>]
```
Add a comment to a section. If `--context` is omitted, the engine captures the current section content automatically. Priority defaults to `normal`, tags default to empty.

```
mdpal comments <bundle> [--section <slug>] [--type <type>] [--unresolved] [--resolved]
```
List comments with optional filters.

```
mdpal resolve <comment-id> <bundle> --response <text> --by <author>
```
Resolve a comment with a response.

```
mdpal flag <slug> <bundle> --author <author> [--note <text>]
```
Flag a section for discussion.

```
mdpal flags <bundle>
```
List all flagged sections.

```
mdpal clear-flag <slug> <bundle>
```
Clear a flag on a section.

```
mdpal diff <rev1> <rev2> <bundle>
```
Diff two revisions within a bundle. `rev1` and `rev2` are version IDs or special values: `latest`, `previous`.

```
mdpal history <bundle>
```
List all revisions in the bundle with version IDs and timestamps.

```
mdpal prune <bundle> [--keep <n>]
```
Prune old revisions. Default keep count from config.yaml. Merges resolved comment history forward before deleting.

```
mdpal version show <bundle>
```
Show the current version ID.

```
mdpal version bump <bundle>
```
Bump the version number (resets revision to 0).

```
mdpal revision create <bundle> [--content <text> | --stdin] [--base-revision <versionId>]
```

Create a new revision from content. Used by the app on explicit save (⌘S): app pipes serialized content via stdin, CLI creates the revision file, updates symlink + pointer file. ## 11. Trade-Offs and Alternatives Considered

### 11.1 Independent Packages vs Direct Library Linking

**Chosen:** Independent packages communicating via CLI commands + ISCP dispatches.
**Alternative:** App links engine library directly (original draft approach).
**Trade-off:** Independent packages add a serialization boundary (CLI JSON) and subprocess overhead. But they provide: (1) clean separation — app and engine evolve independently; (2) equal UX/AX — agents and humans use the same CLI interface; (3) testability — app can mock CLI responses, engine tests don't need UI; (4) pluggability — new document format parsers benefit both app and agents automatically. The overhead is acceptable — local subprocess calls are fast, and the JSON serialization cost is negligible for document-sized payloads. Can tighten for performance later if needed (e.g., shared memory, direct linking) but Jordan explicitly said: "A very pragmatic approach to this implementation."

### 11.1b Protocol Layer (LSP/MCP)

**Deferred.** LSP is line-oriented and assumes the editor owns the document. Our engine is section-oriented. When remote clients need the engine, a protocol adapter can wrap the CLI interface. The CLI JSON format is already a serialization protocol — adapting it to MCP or LSP is straightforward.

### 11.2 Separate Parser Process vs In-Process

**Chosen:** In-process parsing (parser is a Swift protocol, linked directly).
**Alternative:** Parser as a separate process communicating via JSON/protobuf.
**Trade-off:** In-process is simpler, faster, and sufficient for V1. A separate process would enable parsers in other languages (e.g., a Rust-based TOML parser), but that's a future need. The protocol abstraction supports either approach — a future `ExternalParser` could wrap a subprocess.

### 11.3 Document as Class vs Struct

**Chosen:** `class Document` (reference type).
**Alternative:** `struct Document` (value type).
**Trade-off:** The engine holds a mutable document in memory during operations — mutating sections, adding comments, updating metadata. Reference semantics are the natural fit for an object that accumulates mutations before serialization. Value semantics would require either copying the entire document on every mutation or threading `inout` through the call chain. The CLI creates, uses, and discards the document in one pass, but the engine's internal operations still benefit from reference semantics during that pass.

### 11.4 YAML Formatting Preservation

**Chosen:** Reformat on write (clean serialization each time).
**Alternative:** Preserve original YAML formatting (key order, whitespace, quote style).
**Trade-off:** Preserving formatting would require a roundtrip-aware YAML library or manual string manipulation. The metadata block is machine-managed — humans don't hand-edit it. Clean serialization is simpler and avoids subtle formatting bugs.

## 12. Failure Modes

| Failure | Detection | Response |
|---------|-----------|----------|
| File not found | `Document(contentsOfFile:)` throws | CLI: exit 1 with error. App: file open dialog / error alert |
| Unparseable document | Parser throws | `EngineError.parseError`. CLI: exit 1. App: show raw text with error banner |
| Malformed metadata | Yams throws | `EngineError.metadataError`. Engine throws — does not silently discard or overwrite. Caller decides recovery strategy (CLI: exit 1; App: offer to create fresh metadata) |
| Section not found | Slug lookup misses | `EngineError.sectionNotFound` with available slugs for suggestion |
| Version conflict | Hash mismatch | `EngineError.versionConflict` with current content. CLI: exit 2. App: merge/conflict UI |
| Bundle conflict | Symlink changed | `EngineError.bundleConflict` with current document |
| Corrupt symlink | Symlink target doesn't exist | Engine scans for latest revision file by version ID sort, repairs symlink |
| Empty bundle (no revisions) | No `.md` files found | Error. Bundle must have at least one revision |
| Disk full on write | OS error during file write | Propagate error. Revision is not created. Symlink unchanged. No data loss |
| CLI process failure | App receives non-zero exit or crash/timeout | App parses stderr JSON for error details. On timeout/crash, retry or report to user |
| CLI JSON parse error | App receives malformed stdout | App treats as CLI bug. Display raw output in error dialog. Report via dispatch |
| Pointer/symlink mismatch | Engine detects `.mdpal/latest` disagrees with symlink | Engine repairs pointer file from symlink (symlink is authoritative per §6.6) |

## 13. Security Considerations

### 13.1 Input Validation

- **Section slugs:** Validated against `[a-z0-9-/]` pattern. Path traversal attempts (e.g., `../../../etc/passwd`) are rejected.
- **Comment text:** No restrictions on content. Stored as YAML, which handles special characters natively.
- **File paths:** The CLI validates that bundle paths point to actual `.mdpal` directories. Symlink following is limited to the `latest.md` link within the bundle.

### 13.2 No Network Access

The engine has zero network dependencies. No telemetry, no update checks, no external service calls. All operations are local filesystem only.

### 13.3 Bundle Integrity

- Revisions are append-only (new files, never modified in place — except metadata merge-forward during pruning).
- The symlink is the only mutable pointer. It can be reconstructed from the revision list.
- Pruning deletes files but preserves comment history through merge-forward.

### 14.3 Quality Gate Discipline

All tests follow the QG protocol:
- **Bug-exposing test first.** Every bug gets a red test before the fix. No valid test = no valid fix.
- **Red-green cycle.** Write failing test → verify it fails → implement fix → verify it passes.
- **Tests in every iteration.** New functionality = new tests in the same iteration. Not deferred.
- **Phase boundary deep QG.** Full test suite runs at every phase boundary. No regressions allowed.

### 14.4 Test Infrastructure

- **Swift Testing framework** (modern `@Test` macros) for unit and integration tests
- **CLI tests** via `Process` invocation — same mechanism the app uses, validating the real interface
- **Fixture bundles** — pre-built `.mdpal` bundles with known state for deterministic testing
- **No mocks for core engine.** Integration tests hit real parser, real metadata, real file I/O. Mocks only at the CLI process boundary (for app-side testing).

## 15. Phase Sequencing

Phase 1 is **collaborative** — mdpal-cli and mdpal-app work in parallel from day one.

### 15.1 Shared Contract

The CLI command specification (§9) is the interface agreement between components. Both agents develop against this spec:
- mdpal-cli implements the engine core and CLI commands
- mdpal-app builds her scaffold, document model, and CLI integration layer, stubbing CLI responses against the spec
- As CLI commands land, mdpal-app swaps stubs for real `Process` calls

## 16. Open Questions Remaining

These items are noted but do not block implementation. They can be resolved during Phase 1 iteration.

1. **Inline admonitions vs metadata-only comments:** The seed doc shows inline `[!QUESTION]` blocks in the document body alongside metadata block comments. V1 uses metadata block only. Should the engine detect and parse inline admonitions as comments? Deferred — metadata-only is sufficient for V1.

2. **Front matter handling:** The seed doc mentions optional YAML front matter (`---` blocks). Does the engine parse it? Ignore it? The metadata block at the end is canonical. If front matter exists, the engine should preserve it but not interpret it. Deferred to implementation.

3. **Maximum document size behavior:** The PVR says "no hard limits, graceful degradation." What does degradation look like for a 1MB Markdown file? Probably: slower parsing, higher memory usage, but functionally correct. No special handling needed for V1 — just document the expected performance envelope.

4. **Comment ID collision across revisions:** Comment IDs are sequential within a document. If two writers both add comments concurrently (before the conflict is detected), they may assign the same ID. The merge strategy needs to handle this — probably by reassigning IDs during conflict resolution. This is a sub-case of the concurrent write problem (section 7, r002). The sequential ID scheme guarantees collision in concurrent scenarios. The Phase 2 advisory lock (section 7.3) prevents this by serializing writers.
