## 1. Architecture Overview

ISCP is five primitives sharing one notification database and one addressing system:

```
┌─────────────────────────────────────────────────────┐
│                   ISCP v1 Hook                       │
│  (SessionStart + UserPromptSubmit — "you got mail")  │
└──────────────────────┬──────────────────────────────┘
                       │ queries
                       ▼
┌─────────────────────────────────────────────────────┐
│              SQLite DB (~/.agency/{repo}/)            │
│                                                       │
│  ┌──────────┐ ┌───────────┐ ┌────────────────────┐  │
│  │  Flags   │ │ Dispatches│ │   Subscriptions    │  │
│  │ (DB-only)│ │(DB + git) │ │   (DB-only)        │  │
│  └──────────┘ └───────────┘ └────────────────────┘  │
│  ┌──────────────────┐ ┌──────────────────────────┐  │
│  │   Transcripts    │ │      Dropbox items       │  │
│  │  (DB metadata +  │ │   (DB metadata +         │  │
│  │   git payload)   │ │    filesystem payload)   │  │
│  └──────────────────┘ └──────────────────────────┘  │
└─────────────────────────────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │  flag    │ │ dispatch │ │ dropbox  │
    │  (tool)  │ │  (tool)  │ │  (tool)  │
    └──────────┘ └──────────┘ └──────────┘
          │            │            │
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │  /flag   │ │/dispatch │ │ /dropbox │
    │ (skill)  │ │ (skill)  │ │ (skill)  │
    └──────────┘ └──────────┘ └──────────┘
```

**Component relationships:**

| Component | Storage | Payload location | Notification |
|-----------|---------|-----------------|--------------|
| Flag | DB only | N/A (content in DB) | ISCP hook |
| Dispatch | DB notification + git payload | `usr/{principal}/{project}/dispatches/` or `claude/workstreams/{ws}/dispatches/` | ISCP hook |
| Transcript | DB metadata + git payload | `usr/{principal}/{project}/transcripts/` | ISCP hook (via subscriptions) |
| Dropbox | DB metadata + filesystem payload | `~/.agency/{repo}/dropbox/{principal}/{agent}/` | ISCP hook |
| Subscription | DB only | N/A (trigger rules in DB) | Drives other notifications |

**The shared DB is the spine.** Every primitive registers its state in SQLite. The ISCP hook queries one DB to surface all notification types in a single summary line.

### Uniqueness Constraints

```sql
-- Dispatches: one DB record per payload file
CREATE UNIQUE INDEX idx_dispatches_payload ON dispatches(payload_path);

-- Subscriptions: prevent duplicate subscriptions
CREATE UNIQUE INDEX idx_subscriptions_unique ON subscriptions(subscriber, event_pattern, filter);
```

### Indexes

```sql
CREATE INDEX idx_flags_to_status ON flags(to_agent, status);
CREATE INDEX idx_dispatches_to_status ON dispatches(to_agent, status);
CREATE INDEX idx_dispatches_type ON dispatches(type);
CREATE INDEX idx_dispatches_reply ON dispatches(in_reply_to);
CREATE INDEX idx_dropbox_to_status ON dropbox_items(to_agent, status);
CREATE INDEX idx_transcripts_agent ON transcripts(agent);
CREATE INDEX idx_subscriptions_subscriber ON subscriptions(subscriber, status);
CREATE INDEX idx_notifications_to_status ON notifications(to_agent, status);
```

### Schema Versioning

```sql
PRAGMA user_version = 1;           -- Schema version, incremented on every migration
```

The `_iscp-db` library checks `PRAGMA user_version` on init. If the DB exists but has an older version, it runs the appropriate migrations. If the version is newer than the tool expects, it fails with "ISCP DB is from a newer version — update your tools."

### SQLite Configuration

```sql
PRAGMA journal_mode=WAL;          -- Write-Ahead Logging for concurrent access
PRAGMA busy_timeout=1000;          -- Wait up to 1s for locks (within 200ms hook budget + margin for retries)
PRAGMA foreign_keys=ON;            -- Enforce referential integrity
PRAGMA synchronous=NORMAL;         -- Balance durability vs speed (WAL-safe)
```

**Note on busy_timeout:** Set to 1000ms, not 5000ms. The 200ms hook budget means we can't wait 5s. With WAL mode, readers never block, and writer contention is rare (single writer at a time, sub-ms writes). 1s is a generous backstop. If contention regularly triggers the timeout, the architecture has a deeper problem.

## 3. Interfaces

### Tool Layer (bash, `claude/tools/`)

Each tool is a standalone bash script with `_log-helper` integration. Tools handle DB operations directly via `sqlite3` CLI.

#### `flag` (v2)

```
flag <message>                    # flag to self
flag <agent> <message>            # flag to specific agent
flag list [--agent <name>]        # list flags
flag count [--agent <name>]       # count flags
flag discuss                      # pull into /discuss, mark processed
flag clear [--agent <name>]       # clear processed flags
```

#### `dispatch` (v2)

```
dispatch create --to <addr> --subject <text> --type <type> [--priority <p>]
dispatch list [--status <s>] [--type <t>] [--from <agent>] [--to <agent>]
dispatch read <id>
dispatch resolve <id> [--response <dispatch-id>]   # optionally link to a response dispatch
dispatch check                    # quick count for hook use
```

**Payload path resolution:** The DB stores `payload_path` as a path relative to the repo root. On `dispatch read`, the tool resolves the absolute path by finding the main checkout (`git worktree list | head -1 | awk '{print $1}'`) and prepending it. This works from any worktree — the agent never needs to know or construct the main checkout path. The payload is always read from the main checkout's filesystem, which is always up-to-date with master.

**Flag read lifecycle:** `flag list` marks flags as `read` (Slack-style "seen" behavior). `flag discuss` marks them as `processed`. Three states: `unread` → `read` (on list) → `processed` (on discuss/clear).

#### `agent-identity` (new)

```
agent-identity                    # outputs fully qualified agent address
```

Resolves the current agent's identity without relying on environment variables (which don't exist in Claude Code hooks). Resolution chain: (1) detect from git branch (`main/master` → captain, worktree branch name → agent slug), (2) resolve principal from `agency.yaml` via `$USER`, (3) resolve repo from git remote. Cross-validates all sources against `agency.yaml`. Outputs the fully qualified address: `{repo}/{principal}/{agent}`.

**`AGENCY_PRINCIPAL` is deprecated.** The `agent-identity` tool ignores this env var entirely. The mdpal mining found it leaked from the test suite, causing `testuser` resolution. All ISCP tools use `agent-identity` for identity, which validates against `agency.yaml` — never trusts env vars. The `_address-parse` library will be patched to remove `AGENCY_PRINCIPAL` support as part of ISCP Phase 1.

#### `subscribe` (new)

```
subscribe <event-pattern> [--filter <json>]    # register for notifications
unsubscribe <id>                                # remove subscription
subscriptions list                              # show active subscriptions
```

Event pattern grammar (exact match, dot-separated hierarchy):
- `dispatch.created`, `dispatch.resolved` — dispatch lifecycle events
- `flag.created` — new flag
- `phase.complete`, `iteration.complete` — boundary events
- `transcript.available` — new transcript registered
- `dropbox.new` — new dropbox item

Tools that create these entities are responsible for emitting the event (inserting into the `notifications` table for matching subscriptions).

#### `dropbox` (new)

```
dropbox list [--agent <name>]
dropbox fetch <id> --to <destination-path>
dropbox forward <id> --to <agent>          # move item to another agent's dropbox
dropbox audit
```

**Filename sanitization on write:** When a file is registered in the dropbox (by any source), the tool strips path components (`../`, `/`), rejects absolute paths, and stores only the basename. The `filesystem_path` is the actual path in the dropbox directory, constructed by the tool — never taken from external input.

**Destination validation on fetch:** The `--to` destination path must resolve to within the git repo (verified against `git rev-parse --show-toplevel`). Path traversal beyond the repo boundary is rejected.

#### `iscp-check` (new — hook backend)

```
iscp-check [--agent <name>]       # returns summary line or nothing
```

Single tool called by the ISCP hook. Queries all tables in one pass: unread flags, unread dispatches, pending dropbox items, triggered subscriptions. Returns nothing (exit 0, no stdout) when empty. Returns one summary line when items exist.

#### `transcript` (v2)

```
transcript start                  # begin capturing (called by SessionStart hook)
transcript append --role <role>   # append dialogue entry with timestamp (reads from stdin)
transcript decision <text>        # mark a decision inline
transcript summary <text>         # update running summary
transcript end                    # finalize, mark complete in DB
```

**Capture mechanism:** Two-layer approach.

1. **User messages:** The `UserPromptSubmit` hook captures the user's message (available in hook context) and calls `transcript append --role user` piping the message via stdin.

2. **Agent responses:** No `PostAssistantResponse` hook exists in Claude Code. Instead, a hookify rule instructs the agent: "After every substantive response, call `transcript append --role agent` with your actual response — what you said, not a summary of what you said." This is agent-driven, not hook-driven — the hookify rule provides compliance enforcement. The agent pipes its response via stdin to handle multi-line content safely. **Do NOT summarize** — summarizing is what failed in mdpal. Capture the actual dialogue. The ISCP hook includes a staleness check: if no agent transcript entry in the last 5 turns, it warns "transcript capture may be stale."

3. **Decisions and summaries:** The agent calls `transcript decision` and `transcript summary` explicitly during /discuss or when decisions are made — same as today's manual `/transcript` but with tool enforcement.

This means transcript capture is **hook-assisted for user input, agent-driven for agent output, with hookify enforcement for compliance.** It is not fully automatic — the agent must participate. The hookify rule ensures it does. Multi-line content is passed via stdin, not CLI arguments.

### Hook Layer (`claude/hooks/`)

#### `iscp-notify` hook

```json
{
  "hooks": {
    "SessionStart": [
      {
        "type": "command",
        "command": "bash claude/tools/iscp-check"
      }
    ],
    "UserPromptSubmit": [
      {
        "type": "command",
        "command": "bash claude/tools/iscp-check"
      }
    ]
  }
}
```

**Agent identity resolution:** `iscp-check` calls `agent-identity` internally — no `$(...)` subshell in the hook config (which would violate project conventions and trigger permission prompts). On `SessionStart`, `iscp-check` resolves identity and caches it to `~/.agency/{repo-name}/.agent-identity` for the session. On subsequent `UserPromptSubmit` calls, it reads the cached identity — zero fork overhead.

**SessionStart behavior:** When ISCP reports unread items on SessionStart, a hookify rule enforces that the agent's FIRST action is to process them. This is distinct from `UserPromptSubmit` behavior (act at a natural break). SessionStart = act now. Mid-session = act when ready.

#### `transcript-capture` hook

```json
{
  "hooks": {
    "SessionStart": [
      {
        "type": "command",
        "command": "bash claude/tools/transcript start"
      }
    ],
    "SessionEnd": [
      {
        "type": "command",
        "command": "bash claude/tools/transcript end"
      }
    ]
  }
}
```

### Hookify Rules

| Rule | Blocks | Points to |
|------|--------|-----------|
| `hookify.dispatch-manual.md` | Writing any files in `*/dispatches/` without dispatch tool | `/dispatch create` |
| `hookify.flag-manual.md` | Writing to `flag-queue.jsonl` or flag DB directly | `/flag` |
| `hookify.directive-authority.md` | Non-captain sending `directive` type dispatches | Use `request` instead |
| `hookify.transcript-manual.md` | Writing files in `*/transcripts/transcript-*` without transcript tool | `/transcript` |
| `hookify.transcript-capture.md` | Agent must call `transcript append --role agent` after every substantive response | `/transcript` |
| `hookify.dropbox-manual.md` | Direct file manipulation in `~/.agency/*/dropbox/` | `/dropbox` |
| `hookify.session-start-mail.md` | If ISCP reported unread items on SessionStart, agent must process them FIRST | `/dispatch check` |

### Review-Resolution Lifecycle

When a `review` dispatch is sent (e.g., captain reviews agent's code), the resolution cycle is:

1. Captain creates `review` dispatch → payload in git, notification in DB
2. Agent reads it (`dispatch read`) → status changes to `read`
3. Agent evaluates findings, applies red→green fix cycle
4. Agent creates a `response` dispatch with `in_reply_to` pointing to the review dispatch ID → payload contains resolution table
5. Agent resolves the original review (`dispatch resolve <id> --response <response-dispatch-id>`)
6. Captain queries sent dispatches (`dispatch list --from captain --status resolved`) to see resolved reviews

This preserves the audit trail: the review payload is immutable, the resolution is a separate payload linked via `in_reply_to`, both are in git.

### DB Abstraction Layer (`claude/tools/lib/_iscp-db`)

Shared library sourced by all ISCP tools. Handles:

- DB path resolution: `~/.agency/{repo-name}/iscp.db`
- **Repo name sanitization:** Resolved name is validated against `[a-z0-9_-]+` and checked for path traversal (`..`, `/`). GitLab nested group names use the leaf only. The resulting path is verified to be under `~/.agency/` before creation.
- DB initialization (create tables if not exist, set pragmas, check schema version)
- Schema version checking and migration (`PRAGMA user_version`)
- Common queries (unread count, insert, update status)
- SQLite pragma setup (WAL, busy_timeout, foreign_keys)
- SQLite version check (minimum 3.38.0 for `.param set`)
- Named parameter handling (`.param set :name value` / query / `.param clear`)
- Graceful degradation: `iscp-check` handles missing tables without crashing (partial schema state)
- Repo name resolution from `agency.yaml` or git remote

```bash
source "$SCRIPT_DIR/lib/_iscp-db"

# Initialize (idempotent)
iscp_db_init

# Query — uses named parameters via sqlite3 .param set (requires SQLite 3.38+, macOS ships 3.39+)
# The library handles the .param set / query / .param clear cycle internally.
# NEVER interpolate variables into SQL strings — always use iscp_db_query with named params.
iscp_db_query "SELECT count(*) FROM flags WHERE to_agent = :agent AND status = 'unread'" \
  ":agent" "$AGENT"

# Insert — convenience wrappers that use named parameters internally
iscp_db_insert_flag "$FROM" "$TO" "$MESSAGE" "$SESSION" "$BRANCH"

# Update
iscp_db_update_status "flags" "$ID" "read"
```

**Minimum SQLite version:** 3.38.0 (2022-02-22) for `.param set` support. macOS ships 3.39+. Linux distros from 2022+ have this. The `_iscp-db` library checks the version on init and fails with a clear error if too old.

## 5. Technology Choices

| Choice | What | Why | Alternatives considered |
|--------|------|-----|----------------------|
| SQLite | Notification database | No daemon, file-based, concurrent-safe with WAL, ubiquitous | PostgreSQL (overkill, needs daemon), JSON files (no concurrent safety), Redis (needs daemon) |
| Bash | Tool implementation | Framework convention, hooks are bash, `_log-helper` integration | Python (extra dependency), Rust (compilation step), Node (heavy) |
| `sqlite3` CLI | DB access from bash | Ships with macOS/Linux, no additional dependency | Python sqlite3 module (extra runtime), better-sqlite3 node (heavy) |
| WAL mode | SQLite journaling | Allows concurrent readers + one writer, no locking on reads | Journal mode (blocks readers during write), memory (no durability) |
| Filesystem | Dropbox storage | Simple, no format restrictions, works with any file type | DB BLOB (size limits, bad for binary), S3 (overkill, needs network) |
| Markdown | Dispatch payloads, transcripts | Human-readable, diffable, git-friendly, existing convention | JSON (less readable), YAML (parsing complexity), plain text (no structure) |

## 6. Trade-offs

### T-1: SQLite CLI vs embedded library

**Chose:** `sqlite3` CLI invoked from bash.
**Rejected:** Embedded SQLite via Python/Node.
**Reasoning:** Every tool invocation forks a `sqlite3` process. This adds ~10ms overhead per call. An embedded library would be faster but requires a runtime (Python/Node) that isn't guaranteed. The 10ms overhead is negligible against the 200ms budget. Framework convention is bash tools — breaking that for marginal performance gain isn't worth the complexity.

### T-2: Single DB vs per-primitive DBs

**Chose:** Single `iscp.db` with multiple tables.
**Rejected:** Separate `flags.db`, `dispatches.db`, etc.
**Reasoning:** The ISCP hook needs to query all primitives in one pass. A single DB means one connection, one transaction, one query sweep. Multiple DBs would require multiple connections and couldn't do cross-table queries (e.g., "all unread items for agent X").

### T-3: DB-only flags vs git-backed flags

**Chose:** Flags are DB-only, no git payload.
**Rejected:** Flags as files in git (like current JSONL).
**Reasoning:** Flags are quick captures — high frequency, low ceremony. Requiring a git commit per flag defeats the purpose. The DB gives instant capture from any worktree. Trade-off: flags don't survive if `~/.agency/` is deleted. Acceptable — flags are transient by design, processed via `/discuss` then cleared.

### T-4: Dropbox outside git vs inside git

**Chose:** Dropbox at `~/.agency/{repo}/dropbox/` — outside git.
**Rejected:** `claude/dropbox/` inside the repo.
**Reasoning:** Dropbox receives binary files (images, PDFs, exports) that would permanently bloat git history. Files only enter git when the agent routes them to their final destination — with intent and context. The dropbox is a landing zone, not a permanent store.

### T-5: Always-on transcripts vs opt-in

**Chose:** Always-on — every session produces a transcript.
**Rejected:** Opt-in via `/transcript start`.
**Reasoning:** The whole point is Granola-style ambient capture. If it's opt-in, agents will forget (proven by mdpal sessions). The hook starts capture automatically. The cost is one file per session — trivial. The value is complete dialogue history across all sessions.

### T-6: ISCP check on every prompt vs periodic

**Chose:** Check on every `UserPromptSubmit` — every turn.
**Rejected:** Periodic check (e.g., every 60 seconds or every N turns).
**Reasoning:** The SQLite query is indexed and sub-millisecond. The sqlite3 CLI fork is ~10ms. Total cost: ~15ms per prompt — well within the 200ms budget. The simplicity of "always check" beats the complexity of "check sometimes." If empty (99% of checks), zero output tokens — no context cost. The fallback if performance degrades: disable `UserPromptSubmit` hook, keep `SessionStart` only.

## 7. Failure Modes

| Failure | Impact | Mitigation |
|---------|--------|------------|
| DB file corrupted | All notifications lost, tools error | WAL mode + `PRAGMA integrity_check` on init. DB can be rebuilt from git payloads (dispatches still in repo). Flags are transient — loss is acceptable. |
| DB file missing | First run or deletion | `iscp_db_init` creates tables idempotently. Tools self-heal on first invocation. |
| `sqlite3` not installed | Tools fail | Check on first use, clear error message. macOS ships it; Linux needs `sqlite3` package. |
| Concurrent writes | Lock contention | WAL mode + `busy_timeout=5000`. SQLite handles this — one writer at a time, readers never blocked. 5s timeout covers any realistic contention. |
| Disk full | DB writes fail, dropbox writes fail | Tools check for write errors and report clearly. Not much to do — disk full is a system problem. |
| Agent name wrong | Notifications go to wrong agent | Validate against `agency.yaml` on every operation. Never trust `$AGENCY_PRINCIPAL` env var alone (mdpal bug). |
| Hook timeout | ISCP check takes too long | `iscp-check` has a hard 200ms budget. The query is indexed. If it somehow exceeds budget, the hook fails silently — agent continues without notification. |
| Dropbox audit misses items | Stale files accumulate | Audit runs on SessionStart. Captain's audit covers all agents. Items older than configurable threshold flagged. |
| Transcript capture fails | Dialogue lost | Non-fatal — session continues. Transcript is best-effort, not blocking. Agent can still use `/transcript` manually as fallback. |

## 8. Security Considerations

| Concern | Posture | Rationale |
|---------|---------|-----------|
| DB access control | None — filesystem permissions only | Single-machine, single-user trust model (NG-5). The `~/.agency/` directory has user-only permissions (700). |
| Dispatch content | No encryption | Same trust model. Payloads are in git (already readable by anyone with repo access). |
| Dropbox content | No validation | Files from external systems are untrusted but the agent processes them, not executes them. No auto-execution of dropbox items. |
| SQL injection | Parameterized queries only | All `sqlite3` invocations use parameterized inputs via the `_iscp-db` library. Never interpolate user input into SQL strings. |
| Agent impersonation | Sender auto-computed, not self-asserted | Existing convention (DD-7 from dispatch-create). `from_agent` is resolved from git remote + agency.yaml, never from user input. |

## 9. Deployment & Operations

### Installation

ISCP ships as part of The Agency framework. `agency-init` installs:
- Tools to `claude/tools/` (flag, dispatch, dropbox, iscp-check, transcript)
- Library to `claude/tools/lib/_iscp-db`
- Hooks to `claude/hooks/` (iscp-notify, transcript-capture)
- Skills to `.claude/skills/` (flag, dispatch, dropbox, transcript)
- Hookify rules to `claude/hookify/`
- Settings template entries for tool permissions

### First run

`iscp_db_init` creates `~/.agency/{repo-name}/iscp.db` and all tables on first tool invocation. No separate setup step. Directory `~/.agency/{repo-name}/dropbox/` created on first dropbox write.

### Migration

One-time migration tool: `claude/tools/iscp-migrate`
- Reads `usr/{principal}/flag-queue.jsonl` → inserts into `flags` table (deduplicates by message+timestamp)
- Scans `usr/*/*/dispatches/*.md` → creates `dispatches` rows from frontmatter (handles missing fields with defaults, logs warnings for malformed YAML)
- Batch migration only — no lazy "on first read" path. All existing artifacts migrated in one pass.
- Reports migration results: N flags migrated, N dispatches migrated, N warnings, N errors
- Idempotent — uses `INSERT OR IGNORE` with natural keys. Safe to run multiple times.

### Monitoring

- `iscp-check --stats` — DB size, row counts, oldest unread items
- `/dropbox audit` — stale dropbox items
- Tool telemetry via `_log-helper` (existing)
