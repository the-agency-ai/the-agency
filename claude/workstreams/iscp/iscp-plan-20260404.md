## Phase 1: Foundation — DB, Identity, and Flag (the simplest primitive)

Build the infrastructure that everything else depends on, then prove it works with the simplest primitive (flag).

### Iteration 1.1: DB abstraction layer + agent-identity ✓

**Completed:** PVR and A&D with QG. This iteration was the design phase.

### Iteration 1.2: `_iscp-db` library

- Create `claude/tools/lib/_iscp-db` — shared library for all ISCP tools
- DB path resolution with repo name sanitization (`[a-z0-9_-]+`, no traversal)
- Schema creation (all 6 tables, idempotent)
- `PRAGMA user_version = 1` for schema versioning
- SQLite version check (minimum 3.38.0)
- Named parameter handling (`.param set` / query / `.param clear`)
- WAL mode, busy_timeout=1000, foreign_keys=ON
- Graceful degradation for partial schema state
- **Tests:** BATS tests for DB init, parameter safety, version check, path sanitization

### Iteration 1.3: `agent-identity` tool

- Create `claude/tools/agent-identity`
- Resolution chain: git branch → agency.yaml → git remote
- Ignore `AGENCY_PRINCIPAL` entirely (deprecated)
- Cross-validate all sources against agency.yaml
- Output fully qualified address: `{repo}/{principal}/{agent}`
- Cache to `~/.agency/{repo}/.agent-identity` on first call
- **Tests:** BATS tests for master resolution, worktree resolution, cache behavior, detached HEAD fallback
- **Prerequisite fix:** Patch `_address-parse` to remove `AGENCY_PRINCIPAL` dependency

### Iteration 1.4: `flag` tool v2

- Rewrite `claude/tools/flag` — SQLite-backed, agent-addressable
- All subcommands: `flag <msg>`, `flag <agent> <msg>`, `flag list`, `flag count`, `flag discuss`, `flag clear`
- `flag list` marks as `read` (Slack-style seen)
- Three-state lifecycle: unread → read → processed
- UTF-8 support verified
- `/flag` skill updated
- `hookify.flag-manual.md` — blocks writing to old JSONL queue
- **Tests:** BATS tests for all subcommands, agent addressing, UTF-8, concurrent access

### Iteration 1.5: Migration — flags

- Create `claude/tools/iscp-migrate` (flag portion)
- Read `usr/jordan/flag-queue.jsonl` → insert into `flags` table
- Dedup by message+timestamp
- Handle malformed JSONL lines (skip with warning)
- Idempotent (`INSERT OR IGNORE`)
- Report: N migrated, N warnings, N errors
- **Tests:** BATS tests with fixture JSONL files (valid, malformed, duplicates)

**Phase 1 exit criteria:** Flag tool works from any worktree or master. DB created automatically. Agent identity resolves correctly. Old JSONL flags migrated. Enforcement triangle complete (tool + skill + hookify).

## Phase 2: Dispatch — The core primitive

### Iteration 2.1: `dispatch` tool v2

- Rewrite `claude/tools/dispatch` — full lifecycle with DB integration
- `dispatch create --to <addr> --subject <text> --type <type> [--priority <p>]`
  - Validates type against 7-type enum
  - Creates payload in git at resolved location
  - Creates DB notification record (status: `unread`)
  - Tool handles path resolution — agent never manages payload location
- `dispatch list [--status <s>] [--type <t>] [--from <agent>] [--to <agent>]`
- `dispatch read <id>` — resolves payload from main checkout path, marks read in DB
- `dispatch resolve <id> [--response <dispatch-id>]`
- `dispatch check` — quick count for hook use
- **Tests:** BATS tests for all subcommands, type validation, address resolution, cross-worktree read

### Iteration 2.2: Dispatch payload conventions + hookify

- FR-6 naming: `{type}-{slug}-{YYYYMMDD-HHMM}.md`
- Immutable frontmatter (creation-time only)
- `hookify.dispatch-manual.md` — blocks any file creation in `*/dispatches/`
- `hookify.directive-authority.md` — blocks non-captain directive type
- **Tests:** Hookify rule verification

### Iteration 2.3: Review-resolution lifecycle

- Implement the 6-step review-resolution flow
- `dispatch resolve <id> --response <dispatch-id>` links resolution
- Captain outbox: `dispatch list --from captain --status resolved`
- **Tests:** Full lifecycle test: create review → read → fix → create response → resolve with link

### Iteration 2.4: Migration — dispatches

- Extend `iscp-migrate` for dispatch portion
- Scan `usr/*/*/dispatches/*.md` → create DB records from frontmatter
- Handle missing fields (defaults), malformed YAML (warn), old format (no type field → `notification`)
- **Tests:** Fixtures with valid, malformed, and legacy dispatch files

**Phase 2 exit criteria:** Dispatch lifecycle works end-to-end. Captain can create, track, and resolve dispatches. Worktree agents can read dispatches without merging master. Review-resolution audit trail preserved. All existing dispatches migrated.

## Phase 3: ISCP Hook — "You got mail"

### Iteration 3.1: `iscp-check` tool

- Create `claude/tools/iscp-check`
- Single sqlite3 invocation, multi-statement query across all tables
- Calls `agent-identity` internally (reads cache, no fork on warm path)
- Silent when empty (exit 0, no stdout)
- One-line summary when items exist: `📬 2 dispatches, 3 flags`
- < 200ms wall-clock budget (test cold and warm paths)
- Handle missing tables gracefully
- **Tests:** BATS tests for empty, non-empty, partial schema, performance benchmark

### Iteration 3.2: Hook integration

- `iscp-notify` hook: SessionStart + UserPromptSubmit
- `hookify.session-start-mail.md` — enforce processing on SessionStart
- Agent identity caching on SessionStart
- **Tests:** Hook integration test (simulated hook fire)

### Iteration 3.3: Transcript staleness check

- Extend `iscp-check` to track last transcript append per agent
- Warn after 5 turns with no agent entry
- **Tests:** Staleness detection with mock transcript state

**Phase 3 exit criteria:** Agents are notified of unread items on every session start and every turn. Silent when nothing. < 200ms. SessionStart forces immediate processing.

## Phase 4: Dropbox — Universal intake

### Iteration 4.1: `dropbox` tool

- Create `claude/tools/dropbox`
- `dropbox list`, `dropbox fetch <id> --to <path>`, `dropbox forward <id> --to <agent>`, `dropbox audit`
- Filename sanitization on registration (strip paths, reject `..`)
- Destination validation on fetch (must be within repo)
- DB notification for new items
- Delete original after successful fetch
- `/dropbox` skill
- `hookify.dropbox-manual.md`
- **Tests:** BATS tests for all subcommands, filename sanitization, path traversal rejection, forward

### Iteration 4.2: Dropbox audit job

- `dropbox audit` reports stale items across all agent dropboxes
- Configurable staleness threshold
- Captain-appropriate (global view)
- Integration with `iscp-check` — new dropbox items in summary line
- **Tests:** Audit with fixtures (fresh, stale, mixed)

**Phase 4 exit criteria:** Files can be dropped for any agent. Agent notified via ISCP hook. Fetch routes to final git location. Forward works for captain rerouting. Audit catches stale items. Goal state: empty.

## Phase 5: Transcripts — Always-on dialogue capture

### Iteration 5.1: `transcript` tool v2

- Create `claude/tools/transcript`
- `transcript start` — creates transcript file, registers in DB
- `transcript append --role <role>` — reads from stdin, appends with timestamp, **silent** (no stdout)
- `transcript decision <text>` — marks decision inline (silent)
- `transcript summary <text>` — updates running summary (silent)
- `transcript end` — marks complete in DB
- UTF-8 support
- `/transcript` skill
- `hookify.transcript-manual.md` — blocks manual transcript file creation
- `hookify.transcript-capture.md` — enforces agent self-reporting
- **Tests:** BATS tests for all subcommands, stdin handling, multi-line, UTF-8, silent output

### Iteration 5.2: Transcript hooks

- `transcript-capture` hook: SessionStart → `transcript start`, SessionEnd → `transcript end`
- UserPromptSubmit hook captures user message → `transcript append --role user`
- **Tests:** Hook integration test

### Iteration 5.3: Transcript discovery

- DB registration enables cross-agent transcript queries
- `transcript list [--agent <name>] [--date <range>]`
- Integration with `iscp-check` — new transcripts in summary for subscribers
- **Tests:** Cross-agent discovery with multiple transcript records

**Phase 5 exit criteria:** Every session produces a transcript automatically. User messages captured by hook. Agent responses captured by hookify enforcement. Decisions marked inline. Transcripts discoverable across agents.
