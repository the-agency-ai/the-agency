## 1. Problem Statement

### What problem does this solve?

Agency agents cannot reliably communicate across session boundaries. Three specific failures:

1. **No notification layer.** Agents don't know they have unread dispatches, flags, or handoffs until they happen to read the right file. The mdpal bootstrap showed agents saying "Ready." and waiting — they had no signal that work was waiting for them.

2. **Git-coupled messaging breaks in worktrees.** Dispatch payloads live in git on master. Worktree agents can't see them without merging master first. The mdpal-app session used master paths 66 times vs worktree paths 13 times — the #1 friction source across both sessions.

3. **Flags are principal-scoped, not agent-addressable.** The current flag tool writes to `usr/{principal}/flag-queue.jsonl` — one flat queue per principal. No way to flag something for a specific agent, workstream, or cross-repo target.

### For whom?

Agency agents (AI) and principals (human) working across sessions, worktrees, and repositories within The Agency framework.

### Why now?

Agency 2.0 is scaling from single-agent sessions to multi-agent coordination (captain + worktree agents). The mdpal and mock-and-mark workstreams exposed that file-based, git-coupled communication doesn't scale. Every new workstream hits the same friction. ISCP is the foundation that unblocks reliable multi-agent workflows.

## 2. Target Users

| User | Role | Primary interaction |
|------|------|-------------------|
| **Agency agent** | AI instance in a Claude Code session | Receives notifications on startup and at boundaries; sends flags and dispatches |
| **Captain agent** | Coordinating agent on master branch | Sends dispatches (code reviews, directives), receives status updates |
| **Worktree agent** | Feature agent on isolated branch | Receives dispatches, sends flags, reads payloads after notification |
| **Principal (human)** | Directs agent work | Sends flags/dispatches to agents, receives agent flags for /discuss |

## 3. Use Cases

### UC-1: Agent startup notification ("you got mail")

An agent starts a new session. The ISCP hook fires on SessionStart, checks the DB for unread items addressed to this agent, and surfaces a summary: "You have 2 unread dispatches and 3 flags." The agent can then act on them without being told.

### UC-2: Flag capture (agent-addressable)

During work, an agent observes something noteworthy. It runs `/flag "the flag tool doesn't handle unicode"` (self-addressed) or `/flag captain "dispatch-create has a path bug"` (addressed to captain). The flag is stored in the DB immediately — no git commit needed.

### UC-3: Dispatch lifecycle (create → notify → read → resolve)

Captain creates a code review dispatch addressed to the iscp agent. The dispatch payload (markdown) is committed to git at `usr/jordan/iscp/dispatches/`. A notification record is created in the DB. Next time the iscp agent's ISCP hook fires, it sees the notification and reads the payload.

### UC-4: Cross-worktree dispatch propagation

Captain runs /captain-review and dispatches findings to a worktree agent. The payload is committed on master. The worktree agent doesn't need to merge master — the DB notification tells it "you have a dispatch at `usr/jordan/iscp/dispatches/dispatch-foo-20260404.md`" and the agent can read the file from master directly (via git show or main checkout path).

### UC-5: Flag queue processing (/discuss integration)

A principal runs `/flag discuss` to pull all flags addressed to the current agent into a /discuss agenda. The flags are marked as processed in the DB. The 1B1 protocol handles resolution.

### UC-6: Cross-repo dispatch (future)

An agent in the-agency dispatches to an agent in monofolk. The notification includes the repo context. The receiving agent's ISCP hook resolves the cross-repo reference.

### UC-7: Dispatch type: review

Captain reviews a PR branch and creates a dispatch of type `review` addressed to the worktree agent. This replaces the current separate code-review system. The dispatch payload contains findings in the standard format. The agent handles it like any other dispatch — read, evaluate, fix, resolve.

### UC-8: Dropbox — principal intake

Jordan downloads a Figma export and drops it into `claude/dropbox/jordan/captain/`. Next time captain's ISCP hook fires, it sees "📬 1 new dropbox item." Captain runs `/dropbox list`, sees the file, runs `/dropbox fetch` to move it to the appropriate location (seeds dir, workstream, etc.).

### UC-9: Dropbox — external system intake

A Sentry integration drops error reports into `claude/dropbox/jordan/captain/`. A GitHub Actions workflow drops issue attachments. An agent monitoring service drops alerts. Same mechanism — files land in the dropbox, ISCP notifies the agent, agent fetches and routes.

### UC-10: Dropbox — pre-workstream staging

Jordan wants to start a new workstream but hasn't created the agent yet. Drops seed files into `claude/dropbox/jordan/captain/` for the captain. Captain picks them up and uses them when `/workstream-create` runs.

### UC-11: Conversation transcript — always-on dialogue capture

Every session automatically captures the principal↔agent dialogue as a structured transcript. Jordan says something, the agent responds — captured with timestamps. Decisions noted inline as they're made. Summaries build in real time. The transcript is the readable, queryable record of what happened and why — not a machine log of tool calls.

### UC-12: Transcript discovery across agents

The ISCP agent is working on the DB schema. Captain completed a discussion about addressing yesterday. The ISCP agent can discover and read that transcript to understand the decisions and reasoning — not just the handoff summary, but the actual conversation that produced it.

### UC-13: Notification subscriptions

Jordan registers for notifications: "tell me when the ISCP agent completes a phase." mdpal-cli registers: "notify me when a review dispatch is created for my workstream." Agents and principals subscribe to events and get notified through the ISCP hook when those events fire.

## 4. Functional Requirements

### FR-1: SQLite notification database

A SQLite database stored **outside the git repo** at a path relative to the repo root. Stores flag content, dispatch notifications (pointer to git payload), read/unread state, and addressing metadata.

**Decision (2026-04-04):** DB location is `~/.agency/{repo-name}/iscp.db`. Centralized under home directory, outside any repo, no git involvement. `{repo-name}` resolved from `agency.yaml` or git remote origin. SQLite with an abstraction layer for future backend swaps.

### FR-2: Flag tool (v2)

Replace the current JSONL-based flag tool with SQLite-backed, agent-addressable flags.

- `flag <message>` — flag to self (current agent)
- `flag <agent> <message>` — flag to specific agent
- `flag list [--agent <name>]` — list flags, optionally filtered
- `flag count [--agent <name>]` — count flags
- `flag discuss` — pull flags into /discuss agenda, mark processed
- `flag clear [--agent <name>]` — clear processed flags

Flags are DB-only — no git payload. Same addressing scheme as dispatches.

### FR-3: Dispatch tool (v2)

Evolve the current dispatch-create into a full lifecycle tool.

- `dispatch create --to <address> --subject <text> --type <type>` — create payload in git + notification in DB
- `dispatch list [--status <status>]` — list dispatches with DB state
- `dispatch read <id>` — read and mark as read (DB update)
- `dispatch resolve <id>` — mark resolved (DB update)
- `dispatch check` — check for unread dispatches (DB query)

Dispatch types (formal enum, validated on creation):

| Type | Direction | Semantics |
|------|-----------|-----------|
| `directive` | Captain → agent | Authority-based command. Do this. |
| `request` | Agent → agent | Peer-based ask. Do this, please. |
| `review` | Any → any | Structured evaluation of code, design, tests, prose. Generalizes code-review. |
| `notification` | Any → any/broadcast | Informational. Status updates, completions, warnings. Supports narrow and broadcast addressing. |
| `question` | Any → any | Needs an answer. |
| `response` | Any → any | Reply to a question or acknowledgment. Links via `in_reply_to`. |
| `escalation` | Agent → captain/principal | Blocked and needs intervention. Implies urgency and authority-chain routing. Principal always notified. |

Authority enforcement: `directive` restricted to captain-addressed senders (hookify rule). Peers use `request`. Structured evaluations are always `review`, even when replying to a `request` — reply type matches content, not parent. `in_reply_to` handles threading.

### FR-4: ISCP v1 hook

A Claude Code hook that fires on defined events and checks the DB for unread items addressed to the current agent.

- **Trigger events:** SessionStart and UserPromptSubmit (both via Claude Code hooks)
- **Output:** Silent when no unread items (zero tokens). One-line summary when unread items exist (e.g., `📬 2 unread dispatches, 3 flags`).
- **Behavior:** Informational, not interruptive. Agent acts on mail at a natural break, not immediately.
- **Principal trigger:** `/dispatch check` lets the principal explicitly ask "anything waiting?"
- **Token efficiency:** Empty check = no output. Non-empty = one summary line. Never dump full payloads into hook output.

### FR-5: Agent address resolution

Resolve bare, principal-scoped, and fully-qualified addresses to agent identities and payload locations.

- Uses existing Agency addressing hierarchy: `{org}/{repo}/{principal}/{agent}`
- Workstream addressing: `{repo}/{workstream}` → `claude/workstreams/{workstream}/dispatches/`
- Agent addressing: `{repo}/{principal}/{agent}` → `usr/{principal}/{project}/dispatches/`
- Disambiguation: workstream dir check first, then agent registration, then fail

### FR-6: Dispatch payload conventions

- Payloads are markdown files in git at the resolved location
- Immutable once written — no in-place edits
- Naming: `{type}-{slug}-{YYYYMMDD-HHMM}.md`
- Frontmatter (immutable, creation-time only): created, created_by, to, priority, subject, type, in_reply_to. Mutable state (read_by, read_at, resolved_at) lives in the DB only — not in the payload file.

### FR-8: Dropbox (universal intake)

Transient inbound file staging area for content from any non-session source: principals, external systems, other repos, monitoring services. Analogous to a browser's Downloads folder — should always be empty.

- Location: `~/.agency/{repo-name}/dropbox/{principal}/{agent}/` — outside git, alongside the DB
- `/dropbox list` — show pending items for current agent
- `/dropbox fetch` — agent picks up item, places it at final git location (agent decides where), commits it there, original deleted from dropbox
- `/dropbox audit` — report stale items across all dropboxes, notify responsible agents
- DB notification: ISCP hook reports new dropbox items alongside dispatches and flags ("📬 2 dispatches, 1 flag, 3 dropbox items")
- **Goal state: empty.** The agent to which a dropbox item is addressed is responsible for clearing it ASAP.
- **Audit job:** Periodic check for stale items — items sitting unfetched get reported to the responsible agent for action.
- No format restrictions: markdown, images, JSON, whatever the source produces
- No git involvement until the agent commits the file at its final destination — dropbox itself never touches git history

### FR-9: Conversation transcripts (always-on)

Automatic dialogue capture for every session — the Granola model applied to agent↔principal interaction.

- **Always on.** Not opt-in, not manually invoked. Every session produces a transcript.
- **Dialogue only.** What the principal said, what the agent said. Not tool calls, not file contents, not background processing. The conversation.
- **Timestamped.** Every exchange has a timestamp. Correlatable with commits and actions.
- **Real-time summaries.** Decisions captured inline as they're made. Summaries build during the session, not after.
- **Discoverable.** Transcripts are registered in the DB so other agents and future sessions can find them.
- **Location:** `usr/{principal}/{project}/transcripts/transcript-{YYYYMMDD-HHMM}.md`
- **ISCP integration:** DB tracks transcript existence and metadata. ISCP hook can report "new transcript available from captain session" to interested agents.

### FR-10: Notification subscriptions

Agents and principals register interest in events and receive notifications when those events fire.

- `subscribe <event-pattern> [--agent <name>]` — register for notifications
- `unsubscribe <id>` — remove a subscription
- `subscriptions list` — show active subscriptions
- Event pattern grammar (exact match, dot-separated): `dispatch.created`, `dispatch.resolved`, `flag.created`, `phase.complete`, `iteration.complete`, `transcript.available`, `dropbox.new`
- Subscriptions scoped to the subscriber's principal — an agent can only subscribe to events for its own principal's agents or its own workstreams
- Subscriptions stored in the DB. ISCP hook checks subscriptions alongside flags and dispatches.
- Subscriptions evaluated on hook fire, not real-time push. Notifications accumulate and surface on next check.
- Notification output follows the same model: silent when nothing, one-line summary when triggered.

### FR-11: Migration from v1

- Migrate existing JSONL flag queue (`usr/jordan/flag-queue.jsonl`) to SQLite
- Scan all existing dispatch files and create DB records from frontmatter (batch, not lazy)
- Handle malformed frontmatter gracefully (defaults for missing fields, warnings logged)
- No data loss during migration. Idempotent — safe to run multiple times.

## 5. Non-Functional Requirements

### NFR-1: Token efficiency

Hook output must be minimal — a summary line, not full content. Agents read payloads on demand, not on notification.

### NFR-2: No daemon requirement

All operations must work without a running service. SQLite file-based access only. No background process needed.

### NFR-3: Lightweight and fast

Flag capture must be near-instant (< 100ms). Dispatch check must be fast enough for hook use (< 200ms).

### NFR-4: Worktree-safe

Operations must work correctly from any worktree or the main checkout. The DB is shared (outside git), so all worktrees see the same notification state.

### NFR-5: Concurrent-access safe

Multiple agent sessions may access the DB simultaneously. SQLite WAL mode + proper locking. No corruption on concurrent reads/writes.

### NFR-6: Cross-platform

Must work on macOS (primary). Linux support desirable. Windows not required.

### NFR-7: Full UTF-8 support

Flag messages, dispatch subjects, transcript content, and all user-facing text must support full UTF-8: emoji, CJK, RTL, diacritics. SQLite stores TEXT as UTF-8 natively. The `sqlite3` CLI handles UTF-8 correctly when `PRAGMA encoding = 'UTF-8'` (default).

## 6. Constraints

| Constraint | Source | Impact |
|-----------|--------|--------|
| No running daemon | Agency 1.0 failure — fragile agency-service | SQLite file-based only |
| Claude Code hook model | Must integrate with existing hooks (UserPromptSubmit, PostToolUse, etc.) | Hook output size limits, execution time limits |
| Git for payloads | Dispatch payloads must be in git for versioning and cross-agent access | Payload creation requires commit; notification is separate |
| SQLite outside repo | DB must not pollute git history | Path convention: `~/.agency/{repo-name}/iscp.db` |
| Existing address system | Must use Agency addressing hierarchy | No new addressing scheme — extend what exists |
| Token conservation | Agents have limited context windows | Summaries only in hooks; full reads on demand |

## 8. Non-Goals

1. **NG-1:** Real-time chat between agents — ISCP is notification + payload, not a chat protocol
2. **NG-2:** Message streaming or pub/sub — agents check at boundaries, not continuously
3. **NG-3:** GUI or web interface — CLI and hook integration only
4. **NG-4:** Cross-machine communication (v1) — same filesystem only; cross-machine is future
5. **NG-5:** Encryption or access control — trust model within a single machine
6. **NG-6:** Message threading beyond in_reply_to — no conversation trees in v1
