# DevEx — Architecture & Design

## Overview

DevEx builds the commit boundary, test infrastructure, permission model, enforcement tooling, and context economics for TheAgency. This A&D designs the implementation of 10 functional requirements across 3 phases: pre-commit + isolation (Phase 1), Docker full suite (Phase 2), and permission model + enforcement tooling (Phase 3).

The architecture is constrained by Valueflow A&D assignments: §4 (enforcement ladder registry + audit), §6 (QG tiers T1-T4, changed-file scoping, 60s T1 budget), §9 (context budget linter, ships with decomposition).

---

## 1. Architecture Overview

### Component Map

```
commit-precheck (rewritten)
  ├── stage-classifier        — classify staged files by type
  ├── test-scoper             — map changed files to relevant tests
  ├── timeout-runner          — run tests with 60s budget + graceful degradation
  └── stage-hash-verifier     — verify QGR receipt exists for staged changes

test-isolation (new)
  ├── test_helper.bash        — extended with universal isolation helpers
  ├── isolation-guard         — teardown verification (git config, DB, filesystem)
  └── docker-test.sh          — extended to full 32-file suite

enforcement-tooling (new)
  ├── enforcement.yaml        — capability registry
  ├── enforcement-audit       — validates registry against reality
  └── context-budget-lint     — measures @-import token chains per skill

permission-model (updated)
  └── settings-template.json  — audited, comprehensive safe-op permissions
```

### Data Flow: Commit Boundary

```
git commit
  → pre-commit hook
    → commit-precheck
      → stage-classifier: what kind of changes?
        → markdown-only? → allow with warning, skip tests
        → code changes? → continue
      → test-scoper: which tests are relevant?
        → convention map: claude/tools/X → tests/tools/X.bats
        → package fallback: src/foo/ → tests/foo/
        → no mapping? → warn, proceed (don't block on unmapped)
      → timeout-runner: run scoped tests with 60s budget
        → pass? → continue
        → timeout? → warn, allow (graceful degradation)
        → fail? → block commit
      → stage-hash-verifier: does a QGR receipt match?
        → match? → allow
        → no match, markdown-only delta? → allow with warning
        → no match, code delta? → block (re-run QG)
```

### Isolation Layer Architecture

```
Layer 1: In-process isolation (every test, always)
  ├── Fake HOME (prevents DB/cache leaks)
  ├── ISCP_DB_PATH override (explicit test DB)
  ├── GIT_CONFIG_GLOBAL=/dev/null (no global writes)
  ├── GIT_CONFIG_SYSTEM=/dev/null (no system writes)
  ├── BATS_TEST_TMPDIR for all file creation
  └── Teardown guards (hash .git/config, check no debris)

Layer 2: Docker isolation (full-suite runs, T3)
  ├── Read-only repo mount
  ├── Non-root user
  ├── Container destroyed after run
  └── Fallback: in-process with Layer 1 if Docker unavailable
```

## 2. Data Model

### Stage Classification

Staged files are classified into categories that drive test scoping and skip decisions:

| Category | Detection | Action |
|----------|-----------|--------|
| `markdown-only` | All staged files match `*.md` | Skip tests, allow with warning |
| `non-code` | All staged files are non-code (`.md`, `.yaml`, `.json`, `.txt`, dispatches, handoffs, transcripts) | Skip tests, allow with warning |
| `tool-change` | Files in `claude/tools/` | Map to `tests/tools/*.bats` |
| `lib-change` | Files in `claude/tools/lib/` | Map to tests that import the lib |
| `test-change` | Files in `tests/` | Run the changed test files directly |
| `config-change` | `.claude/`, `claude/config/`, `settings*.json` | Warn, no specific tests |
| `app-code` | `.ts`, `.tsx`, `.js`, `.py`, `.rs`, `.go`, `.swift` | Full app test suite |

### Enforcement Registry Schema

```yaml
# claude/config/enforcement.yaml
version: 1

capabilities:
  git-commit:
    level: 5                    # 1=doc, 2=skill, 3=tool, 4=warn, 5=block
    doc: claude/docs/VALUEFLOW.md#commit-workflow
    skill: .claude/skills/git-commit/SKILL.md
    tool: claude/tools/git-commit
    hookify-warn: claude/hookify/hookify.warn-whw-header.md
    hookify-block: claude/hookify/hookify.block-git-commit.md

  mar:
    level: 1
    doc: claude/docs/VALUEFLOW.md#mar

# Per-workstream overrides
workstreams:
  iscp:
    git-commit: 5
    mar: 2
  devex:
    git-commit: 3
    mar: 1
```

### Test Mapping Model

Convention-based mapping with fallback chain:

```
1. Convention: claude/tools/{name} → tests/tools/{name}.bats
2. Package:   claude/tools/lib/{name} → tests/tools/*.bats (grep for import)
3. Manifest:  tool declares test file in header metadata (edge cases)
4. None:      warn "no test mapping for {file}", proceed
```

The manifest format (for edge cases only):

```bash
# Test: tests/tools/my-special-test.bats
```

Single comment line in the tool header. Parsed by test-scoper. Convention handles 90%+ of cases — the manifest is for the exceptions.

## 3. Interfaces

### commit-precheck CLI

```
Usage: commit-precheck [options]

Options:
  --verbose, -v     Show detailed output
  --dry-run         Show what would run without executing
  --skip-tests      Skip test execution (format/lint only)
  --timeout SECS    Override default 60s test timeout
  --version         Show version
  --help            Show help

Exit codes:
  0 = all checks passed
  1 = warnings only (tests skipped, timeout, no mapping)
  2 = blocking failure (test failure, type error)
```

### test-scoper Interface

```bash
# Input: list of staged files (one per line on stdin, or as arguments)
# Output: list of test files to run (one per line on stdout)
# Exit: 0 = tests found, 1 = no tests mapped (not an error)

git diff --cached --name-only | test-scoper
# Output:
# tests/tools/flag.bats
# tests/tools/dispatch.bats
```

### enforcement-audit Interface

```bash
# Validates enforcement.yaml against reality
# Exit: 0 = all valid, 1 = violations found

enforcement-audit [--fix] [--capability NAME]
# Output (structured):
# ✓ git-commit: level 5 — all artifacts present
# ✗ mar: level 2 — missing skill (declared: .claude/skills/mar/SKILL.md)
# 
# 1 violation(s) found
```

### context-budget-lint Interface

```bash
# Measures @-import token chains per skill
# Exit: 0 = all within budget, 1 = over-budget skills found

context-budget-lint [--budget TOKENS] [SKILL_PATH...]
# Default budget: 4000 tokens
# Output:
# ✓ git-commit: 2,340 tokens (58% of 4,000)
# ✗ quality-gate: 5,120 tokens (128% of 4,000) — OVER BUDGET
#
# 1 skill(s) over budget
```

### Universal Isolation Helper Interface

```bash
# In test_helper.bash — available to ALL test files

# Setup: call in setup() of every test file
test_isolation_setup() {
    # Sets: HOME, ISCP_DB_PATH, GIT_CONFIG_*, snapshots .git/config
    # Creates: BATS_TEST_TMPDIR with isolated working space
}

# Teardown: call in teardown() of every test file
test_isolation_teardown() {
    # Verifies: .git/config unchanged, no live DB writes, no working dir debris
    # Cleans: BATS_TEST_TMPDIR
}

# Legacy alias for ISCP tests (backward compatible)
iscp_test_isolation_setup()    → calls test_isolation_setup
iscp_test_isolation_teardown() → calls test_isolation_teardown
```

## 5. Technology Choices

### Language: Bash for tools, BATS for tests

**Why:** The entire tool ecosystem is bash. ISCP tools, test helpers, Docker runner — all bash. Adding a different language for DevEx tools creates a dependency inconsistency. Bash is the right choice for:
- `commit-precheck` rewrite (replacing existing bash)
- `test-scoper` (file classification, grep, path manipulation)
- `enforcement-audit` (YAML parsing via yq or grep, file existence checks)
- Universal isolation helpers (extending existing bash helpers)

**Exception: context-budget-lint.** Token counting requires either:
- **Option A:** `wc -w` approximation (1 token ~= 0.75 words). Simple, no dependencies, ~20% error.
- **Option B:** Python `tiktoken` library. Accurate, but adds Python dependency.
- **Recommendation:** Option A for V2. The budget is 4000 tokens — a 20% error means we warn at ~3200-4800. Good enough to catch egregious violations. Accurate counting is a V3 refinement.

### YAML parsing: grep + awk for enforcement.yaml

**Why not yq?** It's an additional dependency. The enforcement registry schema is simple enough that `grep`, `awk`, and `sed` handle it. If the schema grows complex, introduce yq.

### Test Mapping: Convention over configuration

**Why not a central manifest?** Convention-based mapping (`tools/X` → `tests/tools/X.bats`) covers 90%+ of cases with zero configuration. A central manifest would need maintenance every time a tool is added. The per-tool header comment (`# Test: ...`) handles edge cases without a central file.

## 6. Trade-offs

### T1: Token counting accuracy vs. dependency simplicity

| Option | Pros | Cons |
|--------|------|------|
| `wc -w` approximation | Zero dependencies, fast, simple | ~20% error |
| Python tiktoken | Accurate to Claude's tokenizer | Adds Python dependency, slower |

**Decision:** `wc -w` for V2. The budget enforcement is about catching gross violations (8000-token skill injections), not precision at the boundary.

### T2: Universal isolation vs. opt-in isolation

| Option | Pros | Cons |
|--------|------|------|
| Universal (all tests get isolation automatically via test_helper.bash) | No test can forget isolation. Zero effort per test file. | May break tests that rely on real HOME or git config |
| Opt-in (each test file calls helpers explicitly) | Backward compatible. Test author controls isolation. | 25 test files need manual updates. Easy to forget. |

**Decision:** Universal with opt-out. `test_helper.bash` calls `test_isolation_setup` in its `setup()` by default. Tests that genuinely need real env can `export SKIP_ISOLATION=1` before loading the helper. This makes isolation the default — you have to deliberately choose to bypass it.

### T3: Stage classifier strictness

| Option | Pros | Cons |
|--------|------|------|
| Strict allowlist (only known non-code extensions skip) | No false negatives | New file types need registry update |
| Blocklist (known code extensions trigger, everything else skips) | New non-code types auto-skip | New code extensions could sneak through |

**Decision:** Blocklist approach. The `has_app_code()` pattern already exists in commit-precheck and works well. Extend it: known code extensions trigger tests, everything else is non-code. Safer to miss a new code extension (tests don't run, caught at phase boundary T3) than to block commits on markdown changes.

### T4: QGR enforcement strictness in pre-commit

| Option | Pros | Cons |
|--------|------|------|
| Block commit if no QGR receipt | Forces QG discipline | Breaks manual/principal commits, cold-start scenarios |
| Warn if no QGR receipt | Non-disruptive | Agents can ignore warnings |
| Block for agent commits, warn for principal | Best of both | Requires detecting who's committing |

**Decision:** Warn-only at pre-commit (T1). The git-commit tool (which agents use) does the hard enforcement. Pre-commit is a safety net, not the primary gate. This avoids blocking principals who commit manually.

## 7. Failure Modes

| Failure | Impact | Mitigation |
|---------|--------|------------|
| Test scoper maps to nonexistent test file | Warns, proceeds. No false block. | Convention mapping validated by enforcement-audit |
| `.git/config` corruption despite isolation | All git operations break | Teardown guard fails loudly. Recovery: `git config --unset core.bare`, restore user config from backup |
| Docker not available for T3 | Full-suite can't run in container | Fallback to in-process with Layer 1 isolation. Warn that Docker is preferred. |
| Token counter wildly inaccurate | Over/under-budget skills not caught | `wc -w` is consistently within 20%. Gross violations (2x over) are always caught. |
| Enforcement registry out of sync with reality | False compliance | Audit tool runs at T3 and pre-PR. Catches drift at phase boundaries. |
| Pre-commit timeout kills tests mid-run | Partial test execution | Timeout runner uses SIGTERM + grace period, then SIGKILL. Temp dirs cleaned by teardown. |
| Test creates files outside BATS_TEST_TMPDIR | Working directory pollution | Teardown guard: snapshot working tree (git status) before/after, fail if new untracked files appear in framework paths |

## 8. Security Considerations

### Test Isolation as Security Boundary

Tests running in the live environment are a security risk — they can modify git config (identity spoofing), write to shared databases (data corruption), and create files that get accidentally committed (supply chain).

**Layer 1 (in-process):** Defense in depth. Fake HOME, explicit DB path, git config isolation, teardown guards. Prevents accidental leakage. Does NOT prevent malicious tests (a test could override the env vars).

**Layer 2 (Docker):** Hard boundary. Read-only repo mount, non-root user, container destruction. Prevents both accidental and intentional leakage. Required for T3 (full-suite) runs.

### Permission Model

`settings-template.json` pre-approves read-only operations. The audit categories:

| Category | Operations | Risk |
|----------|-----------|------|
| Read framework | `ls`, `cat`, `git show` on `claude/`, `usr/`, `.claude/` | None — read-only |
| Read ISCP | `sqlite3` on `~/.agency/`, `flag list`, `dispatch list/read` | None — read-only, own data |
| Write sandbox | Files in `usr/{principal}/` | Low — own sandbox |
| Write framework | Files in `claude/tools/`, `claude/config/` | Medium — requires QG |
| Git operations | `git add`, `git commit`, `git log`, `git diff`, `git status` | Low — local only |
| Destructive | `git push`, `git reset --hard`, `rm -rf` | High — NOT pre-approved |

## 9. Deployment & Operations

### Rollout Strategy

**Phase 1 (pre-commit + isolation):** Direct replacement. The new `commit-precheck` replaces the existing one. The new `test_helper.bash` extends the existing one (backward compatible — ISCP aliases preserved). All 32 BATS files updated to use universal isolation.

**Phase 2 (Docker full suite):** Additive. `docker-test.sh` extended to all files. No existing behavior changes. Docker is optional — fallback to in-process.

**Phase 3 (enforcement + permissions):** New tools + config. `enforcement.yaml` is a new file. `enforcement-audit` is a new tool. `context-budget-lint` is a new tool. `settings-template.json` is updated (additive — new permissions, no removals).

### Monitoring / Observability

All DevEx tools integrate with `_log-helper` for telemetry:
- `commit-precheck`: log test scope decisions, timeout events, skip reasons
- `enforcement-audit`: log capability compliance state
- `context-budget-lint`: log per-skill token counts

Telemetry feeds the continual learning loop (Valueflow A&D §10):
- Permission prompt frequency (target: zero) — measured by counting prompts in session transcripts
- T1 gate duration (target: <60s) — measured by commit-precheck telemetry
- Test corruption incidents (target: zero) — measured by teardown guard failures

## 10. Open Technical Questions

### OQ1: Lib-change test mapping

When a file in `claude/tools/lib/` changes (e.g., `_iscp-db`), which tests should run? Option A: grep all `.bats` files for `source.*_iscp-db` and run those. Option B: maintain a reverse-dependency map. Option A is simpler but slow for many test files. Option B needs maintenance.

**Recommendation:** Option A with caching. `test-scoper` builds the reverse map on first run, caches in `BATS_TEST_TMPDIR`. 32 files is small — grep is fine.

### OQ2: Enforcement audit integration point

Where does `enforcement-audit` run? Options: (a) T3 only, (b) T1 as a fast check, (c) standalone on demand. If T1, it must complete within the 60s budget alongside tests.

**Recommendation:** Standalone + T3. The audit is a validation tool, not a commit gate. Run it at phase boundaries and pre-PR, not every commit.

### OQ3: Working directory debris detection scope

The teardown guard needs to detect new files in framework paths. But `git status` is expensive on large repos. Scope the check to specific directories (`claude/agents/`, `claude/workstreams/`, `.claude/agents/`) where test debris has been observed.

**Recommendation:** Targeted `ls` on known debris locations before/after. Cheaper than `git status`, catches the known patterns.
